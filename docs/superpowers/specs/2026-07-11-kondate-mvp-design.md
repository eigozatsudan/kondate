# こんだて日和 MVP 設計書

- 日付: 2026-07-11
- 状態: 敵対的レビュー受入事項統合版・実装前確定
- 対象: 初回公開する実用 MVP
- 改訂: 2026-07-11、忙しい主婦、家族の食事安全、低ITリテラシー・不安定通信、認証、濫用耐性、外部AI境界の敵対的レビューで確定した指摘を反映

## 1. 目的

毎日の献立決めに悩む家庭の利用者が、ITに詳しくなくても、家族構成と冷蔵庫の食材に合った1食分の献立を短時間で決められるWebアプリを提供する。

成功条件は次のとおり。

- スマートフォンで、基本3項目を選ぶだけで献立を生成できる。
- 朝食・昼食・夕食から1食を選べる。
- 家族の年齢層、量、アレルギー、好みを毎回入力しなくてよい。
- 初回設定は最小項目なら約60秒で完了し、途中で離れても再開できる。
- 冷蔵庫にある消費したい食材を献立に活用できる。
- 材料、人数分の分量、手順まで確認でき、そのまま調理へ移れる。
- 指定した調理時間は、複数料理を食卓へ出すまでの献立全体時間として扱われる。
- 献立全体または1品だけを、同じ条件で再生成できる。
- 履歴や再生成には現在のアレルギー・安全条件が必ず再適用される。
- 画面や通信が途中で切れても、生成状態と完成結果を回収できる。
- 無料AIモデルの利用を前提とし、意図しない有料モデル利用を防ぐ。
- 無料AIが利用できないときも、条件に合う候補がある場合は固定の緊急献立を確認でき、候補がない場合は条件を緩めず明示される。
- 認証、個人上限、全体上限によってAIの濫用を抑える。

UI上の製品名には「こんだて日和」を使用する。名称変更は機能設計に影響しない独立したブランディング変更として扱う。

## 2. MVPの範囲

### 含める機能

- Google OAuthとメールのマジックリンクによるログイン
- 認証の送信・確認・期限切れ・再送・キャンセル・元画面復帰
- 家族メンバーの初回設定と編集
- 現行の標準アレルゲン29品目、加工品の本人ラベル確認、年齢別安全制約
- 朝食・昼食・夕食から1食を選んだ献立生成
- 基本条件と任意の追加条件
- 簡易冷蔵庫リストと使い切り食材の選択
- 材料、分量、献立全体時間、統合調理順、構造化した家族向け取り分け
- 献立全体と料理1品の再生成
- 再生成理由、重複結果の拒否、派生案のグループ化と採用状態
- 履歴とお気に入り
- 最終献立と差分同期する編集可能な買い物リスト
- 生成フォームの下書き保存と生成処理の中断復帰
- AIへ送信・保存する家族情報の事前説明
- 無料AI停止時に使う、決定論的に安全条件を照合した固定の15分緊急献立
- 1ユーザー1日5回の成功生成上限
- 成功上限とは独立した利用者ごとの外部AI試行上限と短期レート制限
- アプリ全体のOpenRouter呼び出し上限
- アカウントとユーザーデータの削除
- スマートフォン最適化と基本アクセシビリティ

### MVPに含めない機能

- 1日3食または7日分の一括生成
- 家族間の献立・買い物リスト共有
- 有料プラン、決済、広告
- 厳密な栄養計算、カロリー管理、治療食
- 離乳食、嚥下障害向け食、医療上の食事指導
- 在庫の自動減算、バーコード、レシート読み取り
- 期限通知のプッシュ配信
- 買い物リストのオフライン編集と外部共有
- ネイティブアプリ

## 3. 技術構成

### 3.1 採用技術

- フロントエンド: React、TypeScript、Vite、Tailwind CSS
- ルーティング: React Router
- サーバー状態: TanStack Query
- フォームと実行時検証: React Hook FormとZod
- API: TypeScriptによるNetlify Functions
- 認証・DB: Supabase Auth、Supabase PostgreSQL
- AI: OpenRouter Chat Completions API
- 単体・コンポーネントテスト: Vitest、React Testing Library
- E2E: Playwright
- DBテスト: pgTAP

家族プロフィールや献立条件の途中入力は、認証後にRLSで保護した下書きとしてSupabaseへ逐次保存する。通信断が生成記録作成前に起きても同じ本文を再送できるよう、生成開始時だけ、入力フォーム全体ではなく検証済みのcanonical generation command、idempotency key、直近の生成リクエストID、作成時刻、所有ユーザーIDを端末へ保存してよい。これは30分の固定TTLで、読出し時に30分以上なら即削除し、別ユーザー・破損データは送信せず削除する。ログアウト、アカウント切替、アカウント削除、terminal到達でも必ず消去する。commandには氏名、メール、アレルギー語、prompt、AI応答を入れず、自由変更理由は200文字以内とし個人名を入力しない旨を表示する。認証中だけ必要なPKCE verifier、state、continuation secretはこれと分離した5分の一時領域へ保存し、成功・失敗・期限切れで消去する。

SSRは使用しない。主要画面がログイン後のアプリでありSEO要件がないため、Vite SPAを採用して運用面を単純にする。

### 3.2 本番配置

- Netlify: 静的フロントエンド、Netlify Functions、環境変数
- マネージドSupabase: Auth、PostgreSQL、RLS、マイグレーション
- OpenRouter: Netlify Functionからのみ呼び出す

OpenRouter APIキーとSupabaseの秘密鍵はブラウザへ配布しない。ブラウザにはSupabaseの公開URLとpublishable keyだけを渡す。本番のbrowser/server URLは同じ `https://<project-ref>.supabase.co` というpathなしのmanaged originへ固定し、maintenance direct/session接続も同じproject refへ束縛する。任意HTTPS URL、lookalike、複数projectの混在はbuild前とFunction起動時に拒否する。

### 3.3 ローカル環境

リポジトリ直下のDocker Composeを入口とし、次をまとめて起動する。

- Vite開発サーバーとNetlify Viteプラグインを含むアプリコンテナ
- Supabase公式Docker Compose構成のローカル用サービス群
- マジックリンク確認用のローカルメール受信サービス
- Google成功・キャンセル・拒否を外部アカウントなしで再現する、ローカル専用の決定論的疑似OAuthサービス
- E2E用の疑似OpenRouterサービス

開発者は `docker compose up` を中心にアプリ、Auth、DB、Functionsを確認できる。ブラウザ、Playwright、Supabase redirect、Netlify Functionのorigin検査は、ローカルでは `http://127.0.0.1:5173` だけを正規originとし、`localhost`、別port、別schemeを混在させない。疑似OAuthへの接続設定はlocal/testだけで許可し、本番preflightは疑似provider modeとそのURLを拒否する。ローカルSupabaseのスキーマは本番と同じSQLマイグレーションから構築する。Supabaseの自己ホスト構成はローカル開発専用であり、本番には使用しない。

## 4. システム境界とデータフロー

### 4.1 通常のデータ操作

ログイン済みブラウザはSupabaseクライアントを使い、プロフィール、冷蔵庫、履歴、買い物リストを操作する。ユーザー所有の全公開テーブルでRLSを有効にし、`auth.uid()` と `user_id` が一致する行だけを本人が操作できる。共通の安全カタログは認証済み利用者へ読み取りだけを許可する。

### 4.2 AI生成

1. ブラウザが保留中の自動保存を同期flushし、サーバーが返した下書きの単調増加revisionを含むcanonical commandと一意なidempotency keyを上記の短寿命領域へ保存してからNetlify Functionへ送る。
2. FunctionがSupabase JWT、入力サイズ、列挙値を検証する。同じkeyの既存記録は現在状態を読む前にcanonical request HMACでreplayまたは不一致拒否する。新規keyはDB関数が下書き・元献立・差替料理の所有者と期待revisionを同じトランザクションで検査した後だけ、同時実行状態、利用者外部call attempt日次枠、短期レート制限を予約して `processing` 記録を作る。foreignとmissingは同じclosed errorで、quota行や存在oracleを残さない。失敗記録も後述の保持期間中は削除せず、別キーによる失敗連打を上限判定から逃がさない。
3. DB関数が日本時間の日付で成功候補のユーザー予約枠、最初の外部呼び出しに使う利用者外部call attempt枠、アプリ全体枠を原子的に予約する。
4. Functionが現在の対象メンバー、最新の安全条件、アレルゲン辞書・食品安全ルールのバージョン、献立条件、選択した冷蔵庫食材をDBから読む。
5. AI送信前の決定論的preflightで、対象メンバーが `complete` か、アレルギー・対象外確認が完了しているか、自由登録アレルギーを検査可能か、必須安全制約を適用できるかを確認する。登録アレルゲンとメイン食材・選択した `must_use` / `prefer_use` 冷蔵庫食材の直接・同義語・加工品矛盾、今回だけ避ける食材とメイン・`must_use` 条件の相互矛盾、対象外医療依頼、期限経過食材の今回確認もここで検査する。失敗時はAIへ送らず、未送信の全予約を返す。
6. 外部送信用のallowlist DTOを新たに組み立てる。送信できるのは食事区分、希望条件、短い自由メモ、標準アレルゲンID、確認済み自由登録語、年齢帯・量・希望・必須制約、食材名・量・優先度、再生成に必要な料理内容と変更理由だけとする。user ID、household member ID、pantry item ID、menu/dish/stepのDB UUID、氏名、メールは含めず、メンバー・食材・料理・工程はリクエストごとの `member_1`、`pantry_1`、`dish_1`、`step_1` へ置換する。
7. 過去献立の再生成でも、過去の安全条件で現在の条件を上書きしない。元献立の安定IDも外部へ送らず、同じ一時参照へ変換する。
8. 外部送信の直前に利用者外部call attempt枠とアプリ全体枠を消費済みへ確定し、OpenRouterへ環境変数の優先順どおりに無料モデル一覧を渡す。
9. JSON Schemaによる `success` または `constraint_conflict` の構造化出力を要求する。`success` payloadはリクエスト内だけのlocal refsだけを使い、providerにDB UUIDの生成・反復を求めない。サーバーがlocal refsを型別に解決して全行へ新しいUUIDを割り当てた後に内部献立schemaを検証する。安全処理は料理、匿名メンバー、対象食材、分岐前の料理工程を参照するstructured safety actionとして出力させるが、AIが出したタグやactionだけを安全の証拠にはしない。
10. Zod、アレルゲン辞書、年齢別安全ルール、取り分け工程、冷蔵庫条件、総調理時間の業務ルールで応答を再検証する。料理aggregateのスキーマ上の全text leafを列挙して検査し、structured safety actionが参照する料理・メンバー・工程と、実際の材料・手順本文が一致することを決定論的に確認する。
11. 加工品ラベル確認はサーバーが検査結果からcanonicalに再構築し、すべて `pending` とする。AIが返した `confirmed`、余分な確認、欠落した確認を信用しない。
12. 成功保存トランザクション内で対象メンバー、アレルギー、安全カタログ行をロックし、最新のsafety fingerprintを再計算して、AI送信前の期待値と比較する。不一致なら献立・子行・成功枠を一切確定せず、現在条件での再検査を要求する。
13. fingerprint一致後にだけ、正常な献立、現在の安全条件、希望条件スナップショット、検証バージョン、canonical pending label confirmations、検証済みstructured safety actionsを同じDBトランザクションで保存する。
14. ユーザー成功予約枠と生成記録を `succeeded` へ確定し、結果を返す。

OpenRouterへ送信する前に処理を中止した場合は、成功候補のユーザー枠、未送信の利用者外部call attempt予約、未使用のアプリ全体予約を返却する。OpenRouterへ送信した外部呼び出しは、失敗しても利用者attempt上限とアプリ全体上限から戻さない。

構造または業務ルールに違反した応答は、最初の外部試行が所定時間内に完全な応答を返し、その応答を不正と判定でき、同期Function総予算にも十分な時間が残る場合だけ、実際に応答したモデルを除外した無料モデル一覧で1回自動修正する。修正呼び出しには新しい利用者外部call attempt枠とアプリ全体枠が必要だが、成功候補のユーザー枠は追加予約しない。最初の試行がタイムアウト、通信中断、結果不明、OpenRouter障害になった後は修正を開始しない。修正枠を確保できない場合、修正後も不正な場合、または総予算を満たせない場合は献立を保存せず、成功候補のユーザー予約枠を返却する。

AI応答は `success` または `constraint_conflict` のどちらかとする。選択した冷蔵庫食材、家族の安全条件、料理構成を同時に満たせない場合は `constraint_conflict` と理由、対象条件を返し、献立を保存せず成功候補のユーザー枠を返却する。外部呼び出し済みの利用者attempt枠とアプリ全体枠は消費済みのままとする。

直接一致する登録アレルゲンが残る結果は成功にしない。カレールー、だし製品、しょうゆ、ドレッシング、マヨネーズなど、製品によって原材料が変わる加工品は安全と判定せず、該当する登録アレルギーに対してサーバーが `label_confirmation_required` を導出する。料理名、説明、材料名、分量表記、手順、全体段取り、取り分け量、追加の切り方・加熱・味付け・配膳確認を含むスキーマ上の全text leafを検査する。結果画面では「安全確認済み」と表現せず、AI生成であることと未解決のラベル確認を常時表示する。

### 4.3 生成処理の中断復帰

状態APIは `not_started`、`processing`、`succeeded`、`failed`、`constraint_conflict` の状態を返す。`not_started` は端末がcanonical commandとidempotency keyを保存した後、サーバーが生成記録を作る前に通信が切れた場合を表す。ブラウザは画面表示時、通信復帰時、認証復帰時に、30分以内かつ現在ユーザーへ束縛された端末記録だけを使って状態APIを照会する。

- `not_started`: 保存済み下書きと同じidempotency keyで生成APIを再送する。
- `processing`: 生成中画面へ戻し、同じ生成を再送しない。
- `succeeded`: 保存済みの献立結果へ案内し、「献立ができました」と表示する。
- `failed`: 成功候補のユーザー枠が返却済みであること、成功残数、外部call attempt残数、再試行可能時刻を表示する。
- `constraint_conflict`: 満たせない条件、成功候補のユーザー枠が返却済みであること、送信済みattemptの消費結果を表示する。

通信再試行では同じidempotency keyを使用する。保存後にHTTP応答だけが失われても、生成回数を再消費せず完成結果を回収できる。期限切れの `processing` はDB関数が失敗へ確定し、ユーザー予約枠を返す。

### 4.4 再生成

- 過去の希望条件は再利用するが、現在の対象メンバー、アレルギー、年齢、安全制約、対象外医療条件を必ず上書き適用する。
- 過去の安全条件スナップショットは監査と差分表示にだけ使用し、再生成入力の正としない。
- 履歴表示、買い物リスト化、献立全体の再生成、1品再生成の前に、残す料理を含む献立全体を現在の安全条件で再検査する。
- 履歴の現在安全再検査では、保存済み献立本文と保存時の冷蔵庫表示snapshotを現在の対象メンバー・アレルギー・安全カタログへ照合する。調理後の在庫削除・残量変更、現在の好みの変更だけを過去献立のunsafe/invalid理由にはせず、非阻害の変更詳細として表示する。
- 安全条件が変わっていた場合は対象箇所を表示し、安全確認が終わるまで再生成と買い物リスト化を止める。
- 献立全体: 元の希望条件、現在の安全条件、変更理由を使用する。
- 1品だけ: 元の希望条件、現在の安全条件、残す料理、差し替える料理の役割、変更理由を渡す。
- 再生成の新しいcandidateには現在も存在する対象メンバーだけを含め、削除済みメンバーの取り分け・安全処理・自由文・匿名参照をAI入力にも新versionにも複製しない。現在の冷蔵庫選択と在庫条件は再生成時に別途組み立て、過去の削除済み・数量変更済み在庫を必須条件として黙って再利用しない。
- 全体・1品のどちらも変更理由を必須とし、「もっと簡単」「別の食材」「子ども向け」「別の味」「自由入力」から選べる。`change_reason` は生成記録と献立バージョンへ保存し、AI入力にも渡す。
- 元料理、同じ派生グループ、直近の履歴にある料理名と主要材料構成を除外する。元と同一または実質的に同じ結果は検証失敗とし、利用者の成功回数へ含めない。
- 再生成結果は元データを上書きせず、同じ派生グループ内の新バージョンとして保存する。
- 履歴では派生グループを1件にまとめ、利用者が「これに決めた」と選んだバージョンを代表表示する。
- 全体再生成と1品再生成はいずれも、異なる正常結果が保存された場合だけユーザー上限を1回消費する。
- 買い物リスト作成済みの献立を変更した場合は、材料差分を提示して利用者が同期を承認するまでリストを自動変更しない。

## 5. 認証とアクセス制御

- 未ログイン利用者はログイン画面と認証コールバック以外のアプリ画面へ入れない。
- 認証方法はGoogle OAuthとメールのマジックリンクとする。
- Google OAuthとマジックリンクの認証完了はPKCE code flowを使用する。認証開始は必ずアプリのログイン画面から行い、ブラウザが高エントロピーの `state`、`code_verifier`、一回限りcontinuation secretを生成してから遷移する。`state` はOAuth往復値としてproviderへ渡し、callbackで照合する。verifierとcontinuation secretは開始したブラウザだけに保持し、URLやログへ含めない。callback受領後はcodeとstateをURLから直ちに除去し、アクセスログでは値をredactする。
- callbackは、開始済みで未使用・未期限切れの `state` と認可codeだけを受け付ける。アプリが開始していないcallback、state不一致、再利用、`access_token`・`refresh_token` など未知のtoken fragment、想定外のhash/queryパラメータはセッションへ取り込まず拒否する。
- 登録とログインを別画面にせず、「Googleで続ける」「ログイン用メールを送る」と表現する。
- マジックリンクは「未入力」「送信中」「送信済み」「確認中」「完了」「期限切れ・使用済み」「送信失敗」の状態を持つ。
- 送信済み画面には送信先アドレス、迷惑メール確認、サーバー指定の再送待ち時間、メールアドレス変更、Googleへの切替を表示する。
- 同じブラウザの別タブで認証した場合は、同じローカル認証試行の `state` と verifierを使ってcodeを交換し、初回設定または元の画面へ戻す。元タブは再表示時とSupabase Authイベント受信時にセッションを再確認し、自動復帰する。
- メールアプリ内WebViewなど、元ブラウザとセッション領域を共有しない画面ではセッションを作らない。callbackは短寿命のsingle-use continuationへ認可codeを一度だけ預け、tokenやverifierを表示・保存しない。元ブラウザは端末にだけ保持したcontinuation secretとverifierで結果を一度だけ回収し、そこでcode交換を完了する。WebViewには「ログインを始めたブラウザへ戻ってください」と表示する。元ブラウザを利用できない場合は、そのブラウザで認証を最初からやり直し、WebView単独で続行させない。
- continuationは推測不能な公開IDとsecret hash、開始origin、期限、消費日時だけを保持し、5分で失効する。codeの読出しと交換は一回限りとし、成功・失敗後はcodeとcontinuationを消去する。
- Google認証のキャンセル、拒否、別アカウント選択失敗では、同じログイン画面に再試行、アカウント変更、メールへの切替を表示する。
- セッション切れ後の再ログインでは、認証前の安全な戻り先と保存済み下書きを復元する。
- 匿名ユーザーによるAI生成は許可しない。
- 1ユーザーにつき同時に1件だけAI生成を実行できる。
- すべてのユーザーデータテーブルでRLSを有効にし、別ユーザーの読み書きを拒否する。
- アカウント削除は確認操作後にサーバー側で関連データとAuthユーザーを削除する。

## 6. 家族メンバーと食事上の配慮

単純な家族人数ではなく、複数の家族メンバーを保存する。正確な生年月日は収集せず、安全ルールに必要な年齢帯だけを使用する。

各メンバーは次を持つ。

- 呼び名。任意であり、AIへ送信しない。
- 年齢区分: 離乳食完了後〜2歳、3〜5歳、6〜8歳、9〜12歳、13〜17歳、大人、高齢者
- 食べる量: 少なめ、普通、多め
- アレルギー確認状態: なし、登録あり、未確認
- 標準アレルゲンIDと、確認付きの自由登録アレルギー
- 苦手食材
- 辛さ: なし、控えめ、普通
- 希望する食べやすさ: 指定なし、小さめ、骨なし、やわらかめ。複数選択可
- 必須の安全制約: 骨を除く、小さく切るなど、選択した場合にAIが無視できない項目
- 対象外確認状態: 該当なし、該当あり、未確認
- 対象外種別: 離乳食、飲み込み・むせの不安、医師等から指示された治療食。該当ありの場合に複数選択

初回設定の必須項目は、対象メンバーごとの年齢帯、アレルギー確認状態、対象外確認状態とする。メンバー追加開始時に `draft` 状態の行を作り、項目を選ぶたびに保存する。画面終了や再認証後は未完了の同じメンバーから再開し、完了操作で `complete` にする。進捗と「残りはあとで設定」を表示する。食べる量、苦手、辛さ、食べやすさには年齢帯に応じた保守的な初期値を設定し、最初の献立作成後でも変更できる。

アレルギーが「登録あり」の場合は、消費者庁の現行区分に対応する標準29品目を日本語表示名で検索・複数選択できるようにする。標準候補を実際に表示しないまま「確認しました」と自己申告させない。選択済み項目は名前付きで一覧・削除できる。自由登録では標準候補との一致候補を先に表示し、該当しない場合だけ確認済み文字列と利用者が追加した別名を保存する。

初回完了後も設定画面から、家族の追加・編集・削除、年齢帯、標準・自由登録アレルギー、対象外状態、必須安全制約、量、苦手、辛さ、食べやすさを変更できる。安全条件の変更は新規生成だけでなく、表示中結果、履歴、再生成、買い物リスト化の再検査を直ちに無効化し、最新fingerprintでの再検査を要求する。同一ブラウザeventだけに依存せず、owner-scoped Realtime通知、focus・visibility・online復帰、画面表示中は最大60秒間隔の再確認で別端末変更も検知し、検知から再検査完了まで操作をfail closedにする。

アレルギーが「登録あり」の場合は対象を登録するまで、「未確認」の場合は確認状態を更新するまで、そのメンバーを含むAI献立を生成しない。対象外確認が「未確認」の場合も生成せず、「該当あり」の場合は対象メンバーを外すまで通常献立を生成しない。`draft` 状態のメンバーも生成対象にしない。DBのCHECK制約と生成前検証で、`complete` なのに必須確認状態が欠ける行を許可しない。

標準アレルゲンは表示名だけでなく安定したIDで保存し、同義語・派生原材料の辞書とともにバージョン管理する。自由登録は標準項目との候補照合を表示し、利用者が確認した文字列と別名を保持する。辞書バージョンは献立の検証記録へ保存する。

離乳食完了後の幼児からを対象とする。離乳食、飲み込み・むせに不安がある人、嚥下障害向け食、治療食は生成しない。対象外確認に該当するメンバーを選ぶと生成を停止し、対象メンバーを外すか専門職の指示に従うよう案内する。自由メモに治療食、疾患、嚥下調整を求める内容がある場合も通常生成へ進めない。「やわらかめ」は一般家庭の食べやすさの希望であり、嚥下調整食ではないことを入力時と結果に明記する。

アレルギーと選択した安全制約は必須、苦手食材・辛さ・希望する食べやすさは希望として区別する。安全制約を満たせると検証できない場合は `constraint_conflict` とし、希望条件の未達は結果内で明示する。

生成結果には全体の材料総量に加え、必要な取り分けを構造化して保存する。取り分けは匿名化したメンバー参照、取り分け量、分岐直前の調理工程ID、追加の切り方・加熱・味付け、配膳時の確認を持つ。元の調理工程より後では実行できない取り分け、安全条件を満たさない取り分け、対象が曖昧な取り分けは検証エラーとする。

年齢別食品安全ルールは、少なくとも5歳以下の硬い豆・全種類のナッツ、ぶどう・ミニトマト等の丸く滑りやすい食品と必要な切り方、高齢者の餅等を保守的に扱う。単なる `safetyTags` の存在を根拠にせず、対象料理・匿名メンバー・食材・実行すべき工程を持つstructured safety actionと、材料・手順・取り分け本文の整合を検証する。一般的な「やわらかめ」と嚥下調整食は混同しない。

AI生成レシピだけでアレルギー安全を保証しない。加工品の原材料表示と家庭内の混入を利用者自身が確認する注意書きを、初回設定、生成結果、履歴、買い物リストに表示する。「アレルギー対応済み」「安全」といった保証表現や、安全確認を想起させる緑色チェックは使用しない。未解決の確認は `member_1`、`egg`、DB pathのような内部値ではなく、「○○さん・卵・主菜のマヨネーズ」のように対象メンバー、アレルゲン日本語名、料理・材料・工程を人間向けに表示する。呼び名への対応付けはブラウザまたは本人専用データ内だけで行い、AIへは送らない。

ラベル確認を `confirmed` にできるのは、本人が対象製品の原材料表示を確認して明示操作した場合に限る。本人確認RPCは現在の未解決canonical行だけを、期待する版とともに更新する。AI応答、ブラウザが作った任意tuple、別ユーザー、古い献立版からの更新を受け付けない。買い物リストで別製品を購入する場合は、献立上の過去確認をその製品へ自動継承せず、店頭確認を再び `pending` として表示する。

## 7. 献立作成条件

### 7.1 常に表示する基本条件

1. 食事: 朝食、昼食、夕食
2. メイン食材: 候補チップまたは自由入力。複数指定可
3. ジャンル: 和食、洋食、中華、おまかせ

対象メンバーをメンバーごとのcheckboxで一時選択できるようにし、現在のアレルギー名、未確認状態、対象外状態、必須安全制約を具体的な日本語で常に表示する。「登録アレルギーあり」のような件数・有無だけの要約にしない。生成前に対象を変更または確認できるようにし、安全条件の警告を追加条件の中へ隠さない。生成を止める判定は選択中のメンバーだけを対象とし、対象外のメンバーを一時的に外しても保存済みプロフィールは上書きしない。

### 7.2 折りたたむ追加条件

- 献立全体の調理時間: 15分以内、30分以内、45分以内、指定なし
- 予算: 節約優先、標準
- 今回だけ避ける食材
- 自由メモ。最大200文字とし、命令ではなく献立条件のデータとして扱う。

家族設定は自動反映し、この画面で一時的に対象メンバーを変更できる。一時変更は保存済みプロフィールを上書きしない。自由メモに対象外の治療食・嚥下・離乳食依頼が検出された場合は、AIへ送らず理由を表示する。

入力条件は変更のたびにユーザー専用の下書きへ保存する。画面移動、再読み込み、セッション切れ後も復元し、正常生成が保存された時点で下書きを消す。保存中、保存済み、保存失敗を画面に短く表示し、保存ボタンの連打を必要としない。

### 7.3 生成する品数

- 朝食: 基本2品
- 昼食: 基本2品
- 夕食: 主菜、副菜、汁物の3品

生成結果は献立全体について、食卓へ出すまでの総経過時間と、複数料理を並行して作る統合調理順を含む。総経過時間が指定上限を超える結果は検証失敗とする。

料理ごとには、役割、料理名、説明、個別調理時間、材料、人数分の分量、番号付き手順、構造化した家族向け取り分けを含む。結果画面の料理タブとは別に「全体の段取り」を最初に確認できるようにする。

## 8. 簡易冷蔵庫リスト

冷蔵庫食材は次を保存する。

- 食材名。必須
- 分量と単位。任意
- 期限日。任意
- 期限の種類: 消費期限、賞味期限、その他、不明。期限日を入れた場合でも種類の指定は任意
- 開封状態: 未開封、開封済み、不明。任意
- 作成日時

期限日は利用者入力にもとづく並べ替えと注意表示にだけ使用し、安全性や可食性をアプリが判定しない。入力期限を過ぎた食材は初期状態で献立条件へ追加できず、利用者が実物の状態を今回確認した場合だけ選択可能にする。この確認は可食性の保証として保存・表示しない。

期限経過食材の状態確認は下書きへ保存せず、1つのidempotency keyと日本時間の当日だけで有効な一回限りの確認とする。日付変更、下書き復元、新しい生成、全体・1品再生成では必ず未確認へ戻し、再度の実物確認を求める。

献立作成時に、今回使いたい冷蔵庫食材を複数選択し、それぞれを「必ず使う」または「使えれば使う」にする。メイン食材は料理の中心、「必ず使う」は献立のいずれかで消費する必須条件、「使えれば使う」は未使用理由を表示する希望条件として区別する。アレルギーと矛盾する食材は送信前に拒否する。

数量が入力済みの場合、生成結果は使用予定量、在庫量、不足量を分ける。在庫量以上を必要とする場合、不足分だけを買い物リスト候補にする。数量不明の食材は「在庫量を確認」と表示し、買い物対象から自動除外しない。

生成結果に、各冷蔵庫食材をどの料理でどれだけ使うか、希望食材を使わなかった理由を表示する。必須食材を使えない場合は `constraint_conflict` とする。調理後は「使い切った」「まだある」を片手で選べ、前者は削除、後者は任意で残量を更新する。AIの推測による自動減算は行わない。

## 9. 履歴・お気に入り・買い物リスト

### 9.1 履歴とお気に入り

- 正常生成された献立は自動保存する。
- 新しい順に履歴表示し、再生成の派生バージョンは1つのグループへまとめる。
- グループ内では「これに決めた」バージョンを代表表示し、未選択なら最新の正常結果を表示する。
- 献立単位でお気に入りを付け外しできる。
- 過去条件から全体または1品を再生成できる。
- 過去の履歴は利用者が削除できる。
- 履歴を開くたびに現在の安全条件との差分を確認する。アレルギー、年齢帯、安全制約が変わっている場合は警告を常時表示し、現在条件での再検査が終わるまで「この献立で作る」、再生成、買い物リスト化を無効にする。
- 生成直後を含む献立結果画面も、表示開始時、同一・別タブの家族安全条件変更時、別端末のRealtime通知、focus・visibility・online復帰、最大60秒のperiodic checkで現在条件を再検査する。完了までは古い材料・手順を隠し、ラベル確認、冷蔵庫反映、再生成、採用、買い物操作を無効にする。

### 9.2 買い物リスト

- 献立結果からワンタップで作成する。
- ユーザーごとに「使用中の買い物リスト」を最大1つ持つ。別の献立から作るときは「今のリストへ追加」「新しいリストにする」を選び、確認なしに上書きしない。
- 買い物項目は元になった献立バージョンと料理を保持し、手動追加項目と区別する。
- 野菜、肉・魚、乳製品・卵、乾物、調味料、その他の売り場分類を持つ。
- 同じ正規化名かつ同じ単位で安全に換算できる材料だけを合算する。
- 合算できない単位や曖昧な分量は別項目のまま残す。
- チェック、数量編集、項目追加、削除に対応する。
- 冷蔵庫との照合は正規化名を先に一致させる。名称一致行の数量または単位が不明なら自動減算せず「在庫量を確認」とし、既知の同一単位だけを合算して不足量を求める。異なる単位を推測換算しない。
- 冷蔵庫にある材料を買い物対象から外す「家にある」操作と、誤操作を戻すundoを用意する。数量・単位・売り場も片手で編集できる。
- 元献立を再生成した場合、旧項目と新項目の追加・削除・数量変更、在庫確認状態、発生源、加工品ラベル警告の差分を表示する。利用者が承認した差分だけを反映し、チェック済み項目、手動編集、手動追加、利用者が外した項目を無断で消さない。
- チェック済み・手動編集等のprotected rowと新版に同名・同単位の必要量がある場合、protected rowを残したうえで既購入量との差分を新しい必要量として追加する。候補自体を捨てず、数量を安全に差し引けない場合は別項目で確認を求める。
- 現在の安全条件で再検査し、直接一致するアレルゲンを含む献立からは作成しない。item-levelだけでなく、手順・取り分け等に由来するlist-levelの加工品ラベル警告も、発生源、対象アレルゲン、対象メンバー、辞書版とともに買い物リストへ引き継ぐ。履歴削除後も人間向け発生源snapshotと警告を保持する。
- 使用中リストは初回表示と家族安全条件の変更時に全出典献立を再検査する。別端末変更はowner-scoped Realtime、focus・visibility・online復帰、最大60秒間隔のperiodic checkで検知する。単なるリスト再読込では操作を再開せず、全出典が現在条件で有効かつ同じフィンガープリントである場合だけチェック・編集を許可する。出典削除、再検査不能、invalid、競合時は閉じたまま理由を表示する。不変の作成時警告はread-only provenanceとして残し、操作可否と現在警告はlatest-only projectionだけを正とする。
- 作成・追加・差分反映の完全なcommand、idempotency key、期待list版を個人情報を含めず端末へ一時保存する。DB成功後にHTTP応答が失われた場合は同じkeyと同じrequest hashで再送し、保存済み結果をread backする。別keyで同じ献立版を同じリストへ再追加して項目を二重化しない。

### 9.3 AI不要の緊急献立

アプリ全体上限、利用者外部call attempt上限、短期レート制限、無料モデルの混雑・停止、OpenRouter障害時には、AIを使わない固定の「15分緊急献立」を案内する。

- 固定献立はリポジトリ内の版管理されたデータとし、料理、材料、手順、総調理時間、標準アレルゲンID、加工品ラベル確認、対象年齢帯、structured safety actionsと本文整合を人手でレビューする。
- 表示前に現在の対象メンバーと同じ決定論的安全ルールで絞り込む。
- 直接一致するアレルゲン、対象外の年齢帯、安全制約違反がある候補は表示しない。
- 対象メンバーのアレルギー確認状態が「未確認」、標準カタログへ安全に対応付けられない自由登録アレルギーがある場合、または対象外確認に該当する場合は候補を表示しない。
- 条件を満たす候補がない場合は、無理に献立を表示せず「条件に合う緊急献立がありません」と案内する。
- 緊急献立はAI利用回数を消費せず、AI生成ではないことを明示する。
- AI献立と同じく加工品のラベル確認と家庭内混入の注意を表示し、「安全確認済み」と表現しない。
- 冷蔵庫食材との一致は任意の絞り込みに使うが、安全条件より優先しない。
- 朝食、昼食、夕食の各区分に、対象年齢帯と保守的食品安全ルールを人手レビューした完全な候補を最低1件用意する。各候補は料理構成、材料、人数分の分量、料理別手順、全体段取り、取り分け、加工品ラベル確認を持ち、通常結果と同じくそのまま調理できる画面で表示する。

### 10.2 承認済みの視覚方針

- ホーム: 3ステップ集中型
- 献立結果: 主菜・副菜・汁物を切り替える料理タブ型
- 献立結果の先頭: 総調理時間と全体の段取り
- 背景: 温かいオフホワイト
- 主操作: テラコッタ
- 冷蔵庫関連: 落ち着いた緑
- 本文: 16pxを基準とし、補足文字も12px未満にしない。
- タップ領域: 最低44px相当

料理タブは常時表示し、選択状態を色だけでなく文字、太さ、枠でも示す。アイコン単独の主要操作、スワイプ限定操作、色だけの状態表現は使用しない。

### 10.3 操作上の配慮

- ログイン画面ではGoogleを最上部、メールをその下に表示する。
- マジックリンク送信後は送信先、認証状態、再送までの残り時間、メール変更、Googleへの切替を同じ画面で示す。
- 初回設定は年齢帯とアレルギー確認状態を先に保存し、進捗、「あとで設定」、保存済み状態を示す。
- 初回設定と初回AI生成の直前に「AIへ送る情報」「AIへ送らない情報」「アプリに保存する情報」を平易な3項目で表示する。
- 追加条件は初期状態で折りたたむ。
- 生成ボタンの近くに「本日あとN回」を表示する。
- 「本日あとN回」は `GET /api/usage/today` のサーバー値を正とし、成功残数に加えて外部call attemptの受付可否、短期レート制限中の再開時刻も平易に表示する。端末計算だけで受付可能と断定しない。
- 再生成前に「別の献立が完成した場合に1回使用・現在残りN回」と表示する。
- 生成中は段階的な日本語メッセージと生成IDにもとづく状態を表示し、二重送信を無効化する。根拠のない進捗率は表示しない。
- 生成中に画面を離れても処理は同じIDで追跡し、再訪時に作成中・完成・失敗へ復帰する。
- 保存成功した献立は画面終了でも失われず、ホームに「献立ができました」を表示する。
- 失敗時は成功回数へ含まれないことと、外部送信済みattemptは別上限へ含まれることを区別して表示し、更新後の成功残数・attempt残数・再試行可能時刻を必ず表示する。
- 個人・全体の日次上限は「明日0:00（日本時間）再開」、モデル混雑はOpenRouterの再試行情報があればその時刻、なければアプリの短い再試行待ち時間を表示する。
- AIを利用できないエラー画面には、履歴・お気に入りとAI不要の緊急献立への導線を表示する。
- 現在の安全条件と異なる履歴には画面上部へ警告を固定し、再検査前の主要操作を無効にする。
- AI生成結果には「AIが作成」「加工品はラベル確認が必要」を常時表示し、「安全確認済み」と受け取れる表現を使わない。
- 献立結果の加工品警告には、対象メンバーの呼び名、アレルゲン日本語名、料理・材料・工程を表示し、未確認項目ごとに本人が原材料表示を確認する操作を用意する。内部ref、英語ID、DB pathを利用者向け本文に露出しない。
- 献立結果では各冷蔵庫食材の使用料理と予定量を表示し、調理後に「使い切った」「まだある」を44px以上の片手操作で選べるようにする。「使い切った」は明示確認後に削除し、「まだある」は残量更新を任意とする。どちらもAI推測で自動実行しない。
- 削除は確認または取り消し機能を伴う。
- PCでは中央の最大幅を制限し、スマートフォンと同じ操作順を維持する。

## 11. AIモデル、上限、構造化出力

### 11.1 モデル設定

運営者は環境変数 `OPENROUTER_MODELS` に、優先順のモデルIDをカンマ区切りで指定する。各IDは `:free` で終わり、構造化出力をサポートする必要がある。空、重複、有料IDはローカル起動時と本番ビルド時に拒否する。本番デプロイ検証ではOpenRouter Models APIを参照し、構造化出力への対応も確認する。

OpenRouterの `models` パラメータでモデル間フォールバックを行い、provider設定で要求パラメータ対応を必須にする。無料モデルがすべて利用できない場合、有料モデルへは切り替えない。

### 11.2 上限

- `USER_DAILY_AI_LIMIT`: MVP固定値5。成功した通常生成、全体再生成、1品再生成を合算する。
- `USER_DAILY_EXTERNAL_CALL_LIMIT`: MVP固定値12。利用者ごとにOpenRouterへ実際に送った初回・修正HTTPリクエストを、成功・失敗を問わず合算する。成功生成上限とは独立し、別idempotency keyや失敗で回避できない。
- `USER_SHORT_WINDOW_EXTERNAL_CALL_LIMIT`: MVP固定値4回/10分。利用者ごとの外部送信直前にDBで原子的に判定し、時間窓と再開時刻を返す。
- `GLOBAL_DAILY_AI_LIMIT`: 初期値45。無料枠と公開規模に合わせて運営者が引き下げられるアプリ全体の安全弁とし、NetlifyからOpenRouterへ実際に送ったHTTPリクエストを、成功・失敗を問わず合算する。有料モデルへの切替には使用しない。
- 日付境界: `Asia/Tokyo`
- 同時実行: 1ユーザー1件

成功候補のユーザー枠は予約、成功確定、返却の状態を持つ。利用者外部call attempt枠とアプリ全体枠は予約、外部送信時の消費確定、未送信時の返却を持つ。サーバー停止で予約が残った場合に備え、期限切れ予約をDB関数が安全に解放する。外部送信後に確定した利用者attempt枠とアプリ全体枠は、外部の無料枠消費と濫用耐性を反映するため返却しない。

`failed`、`constraint_conflict`、timeoutを含む生成ledgerは、サーバー秘密鍵でcanonical requestから作るHMAC、利用者、送信回数、終了コード、時刻だけを30日保持する。下書き本文、自由入力、prompt、生AI応答は保持しない。保持期間中の行は日次attempt集計、短期rate limit、idempotency replay、濫用調査に利用し、日次境界を越えたという理由だけで即時削除しない。30日経過後は定期処理で削除し、本人がアカウントを削除した場合は保持期間を待たずcascadeで削除する。

API応答と `GET /api/usage/today` は、今回の成功枠消費、成功残数、利用者外部call attempt残数、短期受付可否、上限種別、既知の場合は再試行可能時刻を含む。全体上限、利用者attempt上限、短期rate limit、OpenRouter障害時も有料モデルへ切り替えず、AI不要の緊急献立を利用可能にする。

### 11.3 出力検証

AI応答はJSON SchemaとZodの両方で検証し、少なくとも次を確認する。

- 食事区分に合う品数と料理役割
- 正の分量または明確な家庭用単位
- 空でない調理手順
- 指定上限以内の献立全体時間と、料理工程を矛盾なく並べた統合調理順
- 「必ず使う」冷蔵庫食材が料理に含まれ、「使えれば使う」の未使用理由があること
- 入力済み在庫量、使用予定量、不足量が矛盾しないこと
- 登録アレルゲンIDと同義語が、料理名・説明、材料名・分量表記、手順、全体段取り、取り分け量・追加処理・配膳確認を含むスキーマ上の全text leafに直接残っていないこと
- アレルギーに関係し得る加工品について、サーバーがcanonicalな `label_confirmation_required` を導出し、AI指定にかかわらず初期状態が `pending` であること
- 現在の対象メンバー、安全条件、アレルゲン辞書バージョンで検証されていること
- 年齢帯ごとの明示的な食品・形状ルールと、選択された必須安全制約を満たすこと
- structured safety actionが対象料理、匿名メンバー、食材、分岐前工程を参照し、材料・手順本文に実際の切り方・骨除去・加熱・配膳確認が存在すること。AIの自由な `safetyTags` だけでは合格にしないこと
- 家族向け取り分けの対象、量、分岐工程、追加処理が元レシピと実行順の上で矛盾しないこと
- 離乳食、嚥下調整、治療食など対象外の依頼を含まないこと
- 再生成時に元料理または同じ派生グループと同一・実質同一の料理でないこと
- 医療・治療効果を断定していないこと

決定論的ルールを通過しても、加工品の製品差、家庭内混入、AI内容の完全な正確性は保証できない。利用者向け注意書きとラベル確認を維持し、検証通過を「安全」と表示しない。

### 11.4 同期Functionの実行時間予算

生成・全体再生成・1品再生成は、Netlifyの同期Function 1回あたり総実行時間を50秒以内に収める。認証、予約、preflight、prompt変換、DB保存を含む総予算であり、クライアント側の待ち時間だけを指さない。

- OpenRouterの各外部試行timeoutは約20秒とし、`AbortController` で打ち切る。
- 最初の試行がtimeout、接続中断、結果不明になった場合はrepairを開始しない。
- repairは、最初の試行が20秒以内に完全応答し、決定論的に不正と判定され、2回目の約20秒と保存・後処理用の余裕を含めても総予算50秒以内と見積もれる場合だけ開始する。
- Functionは単調時計で経過時間を測り、各 `markSent` の前に20秒のprovider予算と2秒のfinalization reserveが残ることを確認する。不足時はHTTPを一度も送らず、当該attemptと全未送信予約を返してterminal timeoutへ確定する。provider timeoutは総deadlineの2秒前を越えず、finalizerも残deadlineでabortされ、50秒後に成功保存を続けない。送信済みattemptだけは返さない。
- timeout後にバックグラウンドで保存やrepairを続けない。状態APIが返すterminal状態とDBのledgerを正とする。

## 12. データモデル

利用者が操作する次のテーブルは `public` スキーマに置く。全行が `user_id` を持ち、RLS対象とする。

- `profiles`: ユーザー単位の初回設定状態
- `household_members`: `draft`・`complete` の設定状態、年齢帯、量、辛さ、食べやすさ、必須安全制約、対象外確認状態・種別、アレルギー確認状態
- `member_allergies`: 標準アレルゲンIDまたは確認済み自由登録
- `member_dislikes`: メンバーごとの苦手食材
- `pantry_items`: 冷蔵庫食材、任意の数量・期限種別・期限日・開封状態
- `generation_drafts`: 自動保存する未送信の献立条件と保存ごとに増えるrevision。生成開始はflush後に返されたauthoritative revisionへ束縛する
- `menus`: 食事区分、ジャンル、希望・安全スナップショット、検証バージョン、派生グループ、親バージョン、変更理由、採用状態、お気に入り
- `menu_target_members`: 生成対象メンバーと生成時の匿名参照
- `generation_pantry_selections`: 冷蔵庫食材、必須・希望、idempotency key単位の状態確認日時、使用状態、使用量、不足量、未使用理由
- `dishes`: 料理の役割、名前、説明、個別調理時間
- `dish_ingredients`: 材料、数量、単位、売り場分類、冷蔵庫食材との対応、ラベル確認フラグ
- `recipe_steps`: 料理ごとの順序付き手順
- `menu_timeline_steps`: 献立全体の時刻・順序・料理工程参照
- `menu_member_adaptations`: 匿名メンバー参照、取り分け量、分岐工程、追加処理、配膳確認
- `menu_safety_actions`: 料理、匿名対象メンバー、対象食材、分岐前工程、必要な切り方・骨除去・加熱・配膳確認と検証結果
- `menu_label_confirmations`: サーバー導出の発生源種別・ID・path・人間向けsnapshot、対象アレルゲンID、匿名対象メンバー参照、辞書・requirement safety fingerprint、現行世代、`pending` または本人RPCで確認した状態・確認日時。安全条件変更時は旧世代を監査用に保持し、現行世代だけを操作対象にする
- `shopping_lists`: 買い物リストの状態、使用中リストの識別、単調増加する版番号、作成時の安全条件フィンガープリント
- `shopping_list_sources`: 買い物リストと献立バージョン・料理の対応、履歴削除後も残す出典snapshot
- `shopping_items`: 分類、材料、数量、チェック、手動項目、手動編集、利用者による除外、在庫確認、ラベル確認参照
- `shopping_item_sources`: 買い物項目と料理・材料の出典、履歴削除後も残す人間向けsnapshot
- `shopping_label_confirmations`: リスト作成・差分承認時点から不変のitem-level・list-level発生源、対象アレルゲン・メンバー・人間向け表示、辞書版、店頭で再確認する状態。履歴削除後のread-only provenanceにも使用する
- `shopping_current_label_warnings`: 使用中リストの現在安全再検査がlist単位で全置換するlatest-only projection。不変の `shopping_label_confirmations` を削除・上書きせず、操作gateだけがこの現行projectionを使用する
- `privacy_consents`: AIへ送る情報の説明バージョンと確認日時

次の安全カタログは `public` スキーマで認証済み利用者へ読み取り専用で公開し、通常のクライアントロールには書き込みを許可しない。

- `allergen_catalog`: 安定したアレルゲンID、表示名、辞書バージョン
- `allergen_aliases`: 同義語、派生原材料、加工品の確認要否
- `food_safety_rules`: 対象年齢帯、食品・形状、安全制約、ルールバージョン

AI・認証制御用の次のテーブルはブラウザに公開しない `private` スキーマに置き、通常のクライアントロールにはGRANTしない。

- `ai_generation_requests`: user_id、idempotency key、canonical requestのサーバーHMAC、種別、`processing` 等の状態、予約、成功・失敗、実モデル、処理時間、元献立参照、完成献立参照
- `ai_user_daily_usage`: user_id、日本時間の日付、予約数、成功数
- `ai_user_daily_external_attempts`: user_id、日本時間の日付、外部call attemptの予約数・送信済み数
- `ai_user_rate_windows`: user_id、短期時間窓、外部call attempt数、再開時刻
- `ai_global_daily_usage`: 日本時間の日付、予約数、外部送信済み数
- `shopping_mutations`: 買い物commandの本人・idempotency key・request hash・responseだけを30日保持するprivate replay ledger。自由入力や安全プロフィールを保存しない
- `auth_continuations`: 未ログイン認証試行の公開ID、continuation secret hash、state hash、開始origin、短い有効期限、一時暗号化した認可code、code受領・消費日時。verifier、token、メールは保存せず、claim成功・失敗・期限切れ時にcodeを消去する

枠の予約・確定・返却は、固定した `search_path` と最小権限を持つDB関数だけから操作する。

生成時の家族条件は監査用の `safety_snapshot`、再利用可能な献立希望は `preference_snapshot` として分離する。プロフィール変更後も過去表示を説明できるよう両方を保存するが、再生成・買い物リスト化・履歴の再利用では、保存済み安全スナップショットではなく現在の安全条件を正とする。

AIの未検証な生応答、内部プロンプト、氏名・メールを含む入力は保存しない。検証済み構造、実モデル、検証ルールのバージョン、サーバー導出の未解決加工品ラベル確認、本文との整合を確認したstructured safety actionsだけを保存する。生成ledgerには自由入力やpromptを保存しない。

## 13. API境界

ブラウザから直接行う通常CRUDはSupabase RLSで保護する。秘密鍵やAIを必要とする処理だけをNetlify Functionへ集約する。

- `POST /api/auth/continuations`: ローカルで開始したPKCE試行のstate hash、continuation secret hash、開始originを短寿命・一回限りで登録
- `POST /api/auth/continuations/:continuationId/callback`: providerから戻った認可codeとstateを一度だけ受け取り、state hash・期限・未使用状態を確認してcodeを短時間だけ暗号化保管。token fragmentは受け付けない
- `POST /api/auth/continuations/:continuationId/claim`: 元ブラウザがsecretとstateを提示し、預けられた認可codeを一度だけ回収。verifierは送らず、元ブラウザでcode交換
- `POST /api/generations/menu`: 新規または献立全体の再生成
- `POST /api/generations/dish`: 1品だけの再生成
- `GET /api/generations/:idempotencyKey/status`: 本人の生成状態と完成献立参照
- `POST /api/menus/:menuId/label-confirmations/:confirmationId/confirm`: 本人が現在のcanonical pending確認を期待版付きで確認済みに変更
- `POST /api/menus/:menuId/revalidate`: 履歴を現在の安全条件で再検査
- `POST /api/shopping-lists/from-menu`: 現在安全条件の再検査と献立由来リストの作成・追加
- `POST /api/shopping-lists/:listId/revalidate`: 使用中リストの全出典を現在安全条件で再検査し、item mutation用の短寿命な現在フィンガープリントを返す
- `POST /api/shopping-lists/:listId/reconcile`: 承認済みの献立変更差分を原子的に反映
- `GET /api/usage/today`: 成功生成残数、利用者外部call attempt残数、短期rate limit・全体上限を含む受付可否と再開時刻
- `DELETE /api/account`: 関連データとAuthユーザーの削除

生成APIはSupabase access token、idempotency key、最新のAI情報説明への同意、Zodで検証可能なJSON本文を要求する。全体・1品の再生成では `change_reason` も必須とする。クライアント指定の安全条件やuser_idを正とせず、現在のプロフィールをDBから読む。外部送信前preflightと成功保存トランザクション内のfingerprint再比較のどちらか一方も省略しない。

自由入力は長さを制限してAI入力内でデータとして明確に区切る。対象外医療依頼を送信前に検出し、AIには外部ツールやデータアクセスを与えず、自由入力にかかわらず構造化出力を再検証してから利用する。OpenRouterへ渡す本文はallowlist DTOのschemaを通した新しいオブジェクトだけから直列化し、元のdraft、DB row、API request全体をspread・埋め込みしない。

状態APIは同じ認証ユーザーのidempotency keyだけを検索し、記録がなければ `not_started`、あれば `processing`、`succeeded`、`failed`、`constraint_conflict`、完成献立ID、枠消費、残り回数、再試行可能時刻を返す。`not_started` の場合だけ端末へ先に保存した新規・全体再生成・1品再生成の完全な同一commandを、kindに対応する生成APIへ同じキーで再送する。端末記録は作成から30分未満かつ現在ユーザー一致の場合だけ読める。既存記録への再送はserver canonical request HMACが一致する場合だけ同じ記録を返し、異なるpayloadは新しい予約やOpenRouter呼び出しより前に拒否する。HMACにはdraft revisionと全再生成条件を含め、同じkeyでmutable draftの後値を読んで別条件を実行しない。

献立由来の買い物リスト項目はブラウザから直接INSERTさせない。作成・追加APIは現在の安全条件を読み、献立全体を再検査し、安全条件フィンガープリントを付けてDB関数へ渡す。DB関数は対象メンバーと買い物リストをロックし、現在のフィンガープリントが一致する場合だけ1トランザクションでリストを作成・追加する。途中でプロフィールが変わった場合は何も書き込まず、再検査を要求する。

差分反映APIは期待する買い物リスト版、元献立版、利用者が承認した追加・削除・数量変更、idempotency keyを要求する。サーバーはitem-levelとlist-level警告を含む差分を再計算する。DB関数は版の一致を確認し、protected rowを保持しつつ不足差分を追加し、出典とcanonical警告を同じトランザクションで反映して版を増やす。作成・承認時の警告snapshotは不変provenanceとして残し、別のcurrent projectionだけを現在安全再検査で入れ替える。通信切断後の同じkey・同じrequest hashによる再送は保存済みの同じ結果を返し、異なるpayloadでのkey再利用を拒否する。

## 14. エラー処理と表示

APIは機械判定用エラーコードと利用者向け日本語メッセージを分離する。

- 個人上限: 「今日は5回利用しました。明日0:00（日本時間）から利用できます」
- 利用者外部call attempt日次上限: 「本日のAIへの送信上限に達しました。献立の成功回数とは別の上限です。明日0:00（日本時間）から利用できます」と緊急献立への導線
- 短期rate limit: 「短時間に続けて送信されました」と具体的な再開時刻、履歴、緊急献立への導線
- 全体上限: 「本日分のAI受付がいっぱいです。成功回数には含まれません。明日0:00から再開します」と緊急献立への導線
- モデル混雑・停止: 「成功回数には含まれません。AIへの送信回数には含まれます」と更新後の成功・attempt残数、再試行可能時刻または待ち時間、緊急献立への導線
- 条件矛盾: 「アレルギー食材が、使いたい食材に含まれています」
- 安全条件変更: 変更された条件と影響する料理を示し、現在条件での再検査を案内する。
- 対象外依頼: 離乳食、飲み込み・嚥下、治療食には対応できないことを示し、通常献立として代用しない。
- 期限経過食材: アプリは可食性を判断しないことを示し、実物の状態確認なしでは必須食材へ追加しない。
- 認証期限切れ: 下書きと戻り先を保持したまま再ログインを案内する。
- マジックリンク期限切れ・使用済み: 同じ画面で再送、メール変更、Google切替を案内する。
- 認証試行不一致・再利用: token fragmentを取り込まず、「このログインは確認できません。ログイン画面からやり直してください」と案内する。isolated WebViewでは元ブラウザへ戻る案内だけを表示する。
- 通信切断: 保存済み生成状態を同じidempotency keyで再確認し、作成中・完成・失敗のいずれかへ復帰する。
- 不正なAI応答: 破損内容を表示せず、成功候補のユーザー枠を返し、送信済みattemptの消費結果と再試行可否を案内する。
- AI timeout: 成功枠は返し、外部送信済みattemptは返さない。timeout後のrepairを行わず、更新後の各残数と緊急献立を表示する。

ログにはリクエストID、エラーコード、処理時間、実際のモデルIDだけを含める。氏名、メール、アレルギー、自由入力、完全なAI入力と出力はログへ残さない。

## 15. テスト方針

### 15.1 単体テスト

- 入力とAI出力のスキーマ
- 無料モデル環境変数の検証
- 過去希望条件と現在安全条件の合成、現在安全条件の優先
- 標準アレルゲン、同義語、派生原材料、加工品ラベル確認の判定
- 加工品ラベル確認の発生源、対象アレルゲン、対象メンバー、辞書バージョンの保持
- 加工品ラベル確認がAI入力にかかわらずserver canonical `pending` になり、余分・欠落・AI指定 `confirmed` を拒否すること
- スキーマ上の全text leafに対するアレルゲン検査。固定文字列を各leafへ1つずつ注入してsource pathも確認すること
- 年齢帯別の食品・形状ルールと必須安全制約。5歳以下の全ナッツ・丸い食品、高齢者の餅等を含むこと
- AIの `safetyTags` だけでは通過せず、structured safety actionの料理・メンバー・食材・工程参照と本文が一致すること
- 離乳食、嚥下・飲み込み、治療食依頼の送信前拒否
- 選択対象メンバーの確認状態、登録アレルゲンとメイン・avoid・`must_use` 食材矛盾をAI送信前に拒否すること
- 取り分け対象、量、分岐工程、追加処理の順序検証
- 献立全体時間と統合調理順の検証
- 冷蔵庫食材の必須・希望、数量、不足、期限確認の判定
- 期限経過食材の確認がidempotency keyと日本時間当日以外へ持ち越されないこと
- 再生成結果の同一・実質同一判定
- 買い物材料の安全な合算
- 名称先行の在庫照合、数量・単位不明時の確認維持、既知同単位在庫の合算
- 献立変更と買い物リスト差分の算出。list-level警告、警告だけの変更、protected rowと新版必要量の差分を含むこと
- 緊急献立の現在安全条件による決定論的絞り込み
- allowlist prompt DTOの組み立てと匿名化。fixtureのuser/member/pantry/menu/dish/step UUIDが直列化本文へ1つも含まれないこと
- 50秒総予算、各試行約20秒、timeout後repair禁止、残予算不足時の送信前中止
- エラーコードから日本語表示への変換

### 15.2 DB・API結合テスト

- RLSが別ユーザーの全CRUDを拒否すること
- PKCEの開始済みstate/verifier、state不一致、再利用、未知token fragment拒否、single-use continuationの期限・一回消費・origin binding
- 日本時間の日次境界
- 同時実行でもユーザー上限を超えないこと
- 個人枠の予約、成功、失敗返却、期限切れ解放
- 利用者外部call attempt日次枠、短期rate limit、アプリ全体枠を同時実行でも超えず、送信済みattemptを失敗後も返さないこと
- failed/constraint/timeout generation ledgerとshopping mutation replay ledgerが30日保持され、別keyによる再試行や応答消失からの回復に使えること。期限後は本文・responseを持たずbounded cleanupで削除されること
- アプリ全体枠を外部送信前の中止では返し、外部送信後の失敗では返さないこと
- idempotency keyによる二重実行防止
- 生成状態APIが本人の処理だけを返し、中断後も同じ結果へ復帰すること
- idempotency key保存後かつ生成記録作成前の切断で `not_started` を返し、同じキーの再送が1件だけ作ること
- 下書きの逐次保存、本人以外の拒否、正常生成後の削除
- `draft` メンバーの項目単位保存、再認証後の再開、必須確認未完了での生成拒否
- 全体・1品再生成で過去希望条件を維持し、現在安全条件を上書き適用すること
- AI応答待ちの間に対象メンバー・アレルギー・安全カタログが変わった場合、成功保存トランザクションのfingerprint比較で全書き込みと成功確定を拒否すること
- 現在安全条件が変わった履歴と残す料理の再検査。調理後の在庫削除・数量変更と好み変更だけでは過去献立をunsafeにせず、削除済みメンバーの文言・参照を新しい再生成結果へ複製しないこと
- 派生献立のグループ化と採用バージョン
- 買い物リストの出典、差分承認、手動変更の保持
- 現在安全条件の再検査と買い物リスト作成の間にプロフィールが変わった場合、全書き込みを拒否すること
- 期待版とidempotency keyによる買い物リスト差分の原子的反映、不変のitem/list-level警告snapshotとlatest-only current projectionの分離、protected数量差分、競合・通信切断・同一payload再送
- 加工品ラベル確認のDB保存、再表示、買い物リストへの継承
- 本人label-confirmation RPCだけが現在版のcanonical pending行をconfirmedにでき、AI・ブラウザ任意tuple・別ユーザー・旧版を拒否すること
- AI情報説明のバージョンと確認状態
- アカウント削除の関連データ削除

### 15.3 コンポーネント・E2Eテスト

- ローカル疑似OAuth/メールプロバイダによるGoogle成功・キャンセル、PKCE state不一致、未知fragment、マジックリンク送信済み・同一ブラウザ別タブ完了、isolated WebViewからsingle-use continuationを経た元ブラウザ完了、期限切れ・再送・メール変更
- 必須項目だけの初回家族設定、標準29品目の検索・複数選択・削除、自由登録候補照合、最初のメンバーの項目途中保存、1人ごとの完了、途中離脱と再認証後の復帰
- 初回後の家族・アレルギー編集、対象メンバー一時選択、選択メンバーだけの生成停止、具体的なアレルゲン名・必須制約の安全要約
- AIへ送る・送らない・保存する情報の説明と未確認時の生成拒否
- 献立下書き、冷蔵庫の必須・希望選択、期限経過時の状態確認
- 日付変更、下書き復元、新規・再生成時の期限経過食材確認リセット
- 生成中の画面終了、通信切断、タブ破棄、応答消失後の状態・結果回収
- idempotency key保存直後かつ生成POST受理前のタブ破棄と `not_started` からの再送
- 総調理時間、全体の段取り、料理タブ、構造化した取り分け
- 人間向けの対象メンバー・アレルゲン・発生源表示、本人ラベル確認、調理後の「使い切った」「まだある」と任意残量更新
- 再生成理由、同一結果の非消費、派生履歴、「これに決めた」
- 献立全体と1品の両方で変更理由が保存・AI入力・履歴へ反映されること
- 安全条件変更後の履歴警告、再検査、操作停止。同一端末event、別端末Realtime、focus・visibility・online復帰、最大60秒periodic checkの各経路を含むこと
- 買い物リスト作成、追加・新規選択、数量編集、「家にある」・undo、献立変更差分、list-level警告、protected数量差分、手動編集保持
- 買い物リスト作成直前の安全条件変更、作成・差分反映の応答消失、同じkey/payloadのreadback、版競合、同一献立版の二重追加防止
- 冷蔵庫希望食材の未使用理由が保存後の再表示でも維持されること
- 成功上限、利用者外部call attempt日次上限、短期rate limit、全体上限、モデル混雑、残り回数、再開時刻、朝昼夕の完全な緊急献立
- 空状態、対象外医療依頼、削除、認証期限切れ
- 主要なスマートフォン幅とPC幅
- キーボード操作、フォーカス、ラベル、コントラストの自動検査

通常テストは疑似OpenRouterと決定論的なローカルOAuth/メールプロバイダを利用し、外部無料枠や実Googleアカウントを消費しない。PKCE、state、continuation、callback成功・失敗は通常CIで自動化する。実OpenRouterのスモークテストは明示的な環境変数がある場合だけ限定CIで1回実行する。

実Google OAuthの成功だけは、Google側画面・アカウント・bot対策を決定論的CIへ固定しない例外とする。リリース候補ごとにstagingの許可済みoriginで1回確認し、候補commit SHA、stagingが実際に配信したdeploy SHAの権威あるreadback、staging origin、実施日時、有効期限、実施者、開始画面、state一致、元ブラウザでのcallback完了、tokenを含まない結果を外部のrelease証跡へ残す。release時点で24時間以内かつ未期限切れで、候補SHA、staging deploy SHA、ローカルHEADが完全一致する証跡だけを許可する。本番deploy後もdeploy IDをNetlify APIから読み戻し、metadataの `commit_ref`、deploy origin、candidate SHA、release tagを一致させ、そのmetadata由来originだけをsmoke対象にする。キャンセル、拒否、state不一致、未知fragment、isolated WebView、再利用、期限切れは例外にせず自動テストする。

モデル採用・優先順変更時には、プロフィール更新、直接・同義アレルゲン、加工品、年齢別食品形状、対象外医療依頼、取り分け工程、在庫不足、重複再生成を含む固定の敵対的応答セットを必ず実行する。

### 15.4 CIゲート

静的解析、型検査、単体、DB、結合、E2E、Netlify向け本番ビルドを順に実行する。いずれかが失敗した変更はデプロイ対象にしない。本番デプロイ後はauthoritative metadataで同じcandidate SHAとoriginを確認してから、そのoriginの公開ログイン画面と認証必須APIの拒否応答を確認する。リリース承認では通常CIの合格に加え、同じcommit SHAに対する期限内の実Google staging成功証跡を要求し、証跡へcode、token、verifier、メールを記録しない。

## 16. セキュリティとプライバシー

- OpenRouter APIキーとSupabase秘密鍵をサーバー環境変数だけで扱う。
- RLSと最小権限のGRANTをすべての公開テーブルに適用する。
- Netlify FunctionでJWTを再検証し、クライアント送信のuser_idを信用しない。
- PKCE、ローカル開始済みstate/verifier、短寿命single-use continuationで認証callbackを開始ブラウザへ束縛し、未知token fragmentを拒否する。
- idempotency、同時実行制限、成功上限、利用者外部call attempt日次上限、短期rate limit、全体上限、30日間の非機密failed ledgerで濫用を抑える。
- 自由入力を短く制限し、AIプロンプトでは明確にデータとして分離する。
- OpenRouter入力は専用allowlist DTOだけから構築し、元のrequest、draft、DB rowを直列化しない。家族の呼び名、メール、正確な生年月日、user ID、household member ID、pantry/menu/dish/stepの安定UUIDをAIへ送らない。
- 秘密情報や食事制約をアプリログへ出さない。
- 初回設定と初回AI生成前に、AIへ送る年齢帯・量・アレルギー・希望条件、送らない氏名・メール、アプリに保存する履歴・条件を平易に説明する。
- OpenRouterと、フォールバックにより実際の無料モデル提供者が変わり得ることを説明し、運営者のプライバシー説明へリンクする。
- 説明文をバージョン管理し、未確認または更新後未確認の場合はAI生成を開始しない。同意しない利用者もAI不要の緊急献立を閲覧できる。
- AIへ送るメンバーは `member_1` のような一時的参照に置き換え、DBのメンバーIDも送らない。
- AIが返したlabel confirmation statusや `safetyTags` を権限・安全判定の根拠にしない。canonical pending確認、structured safety action、本文整合はサーバーが導出・検証し、confirmedへの変更は本人RPCだけに限定する。
- 現在の安全条件を再生成、履歴再利用、買い物リスト化で必ず再適用し、古いスナップショットを安全判断に使用しない。
- アカウント削除と履歴削除を利用者自身が実行できる。

## 17. 受け入れ条件

MVP完成は、次をすべて満たした状態とする。

1. Docker Composeからローカルアプリ、Supabase、メール確認、決定論的な疑似OAuth・疑似AIを起動できる。
2. GoogleとマジックリンクはPKCEとローカル開始済みstate/verifierでログインでき、未知token fragment、state不一致、再利用を拒否する。同一ブラウザ別タブでは同じ試行へ復帰し、isolated WebViewでは短寿命single-use continuationを経て元ブラウザだけがcode交換を完了する。期限切れ、再送、キャンセル、元ブラウザを失った場合も安全にやり直せる。
3. 年齢帯、アレルギー確認状態、対象外確認状態だけなら約60秒で初回設定でき、標準29品目を検索・選択できる。最初のメンバーの項目途中、1人ごとの完了、再認証後から復帰でき、初回後も家族・アレルギー・安全条件を追加・編集・削除できる。
4. 初回AI生成前に、AIへ送る・送らない・保存する情報を確認でき、未確認ならAI生成されない。OpenRouterへ送るallowlist DTOに氏名・メールとuser/member/pantry/menu/dish/stepの安定UUIDが含まれない。
5. 朝昼夕、メイン食材、ジャンル、対象メンバーから献立を生成できる。対象メンバーごとの具体的アレルゲン・必須制約を生成前に確認でき、未確認・対象外メンバーだけを一時的に外せる。対象メンバー安全状態とアレルギー・`must_use` 矛盾はAI送信前に拒否される。
6. 最新のアレルギー、安全条件、アレルゲン辞書、年齢別ルールが新規・履歴・再生成・買い物リストの全経路で適用される。AI応答中に条件が変わった場合も、成功保存トランザクション内のfingerprint再比較で古い結果を一切保存しない。
7. スキーマ上の全text leafを検査して直接アレルゲンを含む結果を保存しない。加工品確認はserver canonical `pending` とし、本人RPCだけが現在版をconfirmedにできる。発生源、アレルゲン日本語名、対象メンバーを材料・手順・取り分け・結果・買い物リストへ人間向けに表示する。AIの `safetyTags` を証拠にせず、幼児の全ナッツ・丸い食品、高齢者の餅等をstructured safety actionと本文整合で検証する。
8. 対象外確認が未確認または該当ありの選択メンバーを含めず、離乳食、飲み込み・嚥下、治療食の依頼を通常献立として生成しない。
9. 冷蔵庫食材を「必ず使う」「使えれば使う」で選び、数量、使用先、不足、保存後も再現できる未使用理由を扱える。期限経過時の状態確認は同じidempotency keyとJST当日だけ有効で、日付変更、下書き復元、新規・再生成へ持ち越されない。調理後は「使い切った」「まだある」と任意残量更新を明示操作できる。
10. 献立全体の指定時間以内で、総調理時間と複数料理の統合調理順を確認できる。同期Functionは総予算50秒以内、外部試行は各約20秒で、timeout・結果不明後にrepairしない。
11. 材料、人数分の分量、手順、対象・量・分岐工程を持つ家族向け取り分けを料理タブで確認できる。ラベル確認と冷蔵庫使用先は内部refでなく家族・料理・材料名で理解できる。
12. idempotency key保存直後、生成中、保存後応答前のいずれで画面終了・通信切断が起きても、`not_started` を含む状態と完成結果を同じkeyで回収でき、成功枠・外部attempt・OpenRouter呼び出しを二重消費しない。
13. 変更理由を指定して献立全体または1品を再生成でき、理由が保存・匿名化AI入力・履歴へ反映され、同一結果は成功回数に含まれない。残す料理も現在安全条件で再検査する。
14. 再生成の派生案が履歴でまとまり、「これに決めた」献立を識別できる。履歴を開くたび、別端末の変更を含む最新条件を最大60秒以内または復帰・Realtime受信時に自動再検査し、完了まで調理・再生成・買い物操作を止める。調理後の在庫残量変更だけで履歴をunsafeにせず、再生成には現在も存在する対象メンバーだけを含める。
15. 現在安全条件の再検査と買い物リスト作成を競合なく行い、作成・差分反映の応答消失から同じkey/payloadで回復できる。使用中リストも表示開始・同一/別端末の家族変更・復帰・最大60秒periodic checkで全出典を再検査し、再読込だけでは操作を再開せず、削除済み出典・invalid・競合時はitem mutationを拒否する。差分は人間向けの各変更と個別承認、不変のitem/list-level警告provenance、別管理のcurrent warning projection、protected rowの不足数量、在庫確認を含めて原子的に反映し、チェック済み・手動編集・手動追加を失わない。名称先行の在庫照合、数量・単位・売り場編集、「家にある」・undo、同一献立版の二重追加防止が動作する。
16. 履歴、お気に入り、下書き、買い物リスト、label confirmation、認証continuationが本人または開始ブラウザにだけ保存・消費され、別ユーザーから読み書きできない。
17. 成功生成は1日5回、利用者外部call attemptは1日12回・10分4回、アプリ全体外部送信は所定上限に制限され、同時実行で超過しない。failed generation ledgerとshopping mutation replay ledgerを30日保持し、その後bounded cleanupする。別keyによる回避を防ぎ、`GET /api/usage/today` と失敗画面で各残数、受付可否、再開時刻を確認できる。
18. 利用者attempt・短期・全体上限または無料モデル障害時に有料モデルへ切り替えず、現在条件に合う朝食・昼食・夕食の完全な固定献立を材料・分量・手順・段取り・取り分け付きで表示する。候補がなければ条件を緩めず明示する。
19. `:free` 以外のOpenRouterモデル設定で起動またはデプロイ検証が失敗する。
20. AI失敗・条件矛盾・不正応答・timeout・保存直前の安全条件変更では壊れた献立を保存せず、成功候補枠と未送信予約を返す。送信済み外部attemptは返さず、timeout後にrepairしない。
21. 主要スマートフォン幅で横スクロールせず、主要操作が44px以上の領域を持ち、料理タブ、dialog、エラー、非同期状態をキーボードと支援技術で操作・理解できる。
22. CIの型、単体、DB、結合、E2E、固定敵対テスト、疑似OAuth、ビルドがすべて成功する。同じcommit SHAについてtoken等を含まない実Google OAuthのstaging成功証跡があり、本番deploy IDのNetlify metadata readbackでもcandidate SHA・release tag・production originが一致した場合だけリリースできる。

## 18. 運用上の前提

OpenRouter無料モデルは提供状況と上限が変動する。運営者はデプロイ時に、構造化出力へ対応する現行の無料モデルIDを `OPENROUTER_MODELS` に明示する。モデル停止時は環境変数の順序を更新する。有料モデルを自動追加する処理は実装しない。

標準29アレルゲン、同義語・派生原材料・加工品、幼児と高齢者を含む食品形状ルールは版管理し、公的情報の変更時とリリース前にレビューする。ルール更新後は履歴を新バージョンで再検査できるようにし、古い検証結果を安全確認済みとして扱わない。

朝食・昼食・夕食の固定緊急献立は追加・変更時に、料理、材料、分量、手順、段取り、取り分け、加工品、対象年齢帯について人手レビューと決定論的安全テストを必須とする。該当する安全な候補がない利用者へ、条件を緩めて表示しない。

利用者外部call attempt日次上限またはアプリ全体上限に達した場合は受付終了を表示し、翌日の日本時間に自動再開する。短期rate limitは具体的な再開時刻を表示する。OpenRouter内部のモデル・プロバイダーフォールバックが複数の無料試行として扱われる場合、アプリが数える1回のHTTP呼び出しより先にOpenRouter側上限へ達する可能性がある。その場合も課金へ切り替えず、モデル混雑・停止として案内する。

生成ledgerとshopping mutation replay ledgerの30日削除、失効した認証continuationの削除、期限切れ予約の解放は、専用maintenance loginによるカテゴリごとのbounded定期ジョブで実行し、件数と終了コードだけを監視する。自由入力、認可code、token、verifierを運用ログへ出さない。同期Functionのp95と最大実行時間、外部試行timeout、timeout後repairが0件であることを監視し、50秒予算に近づいた場合はモデル追加ではなく処理を送信前に止める。

公開規模が無料枠を継続的に超える場合は、有料モデルへの自動移行ではなく、運営方針を改めて決定してから別スコープで設計する。

AIへ送る情報、OpenRouter、利用する無料モデル提供者、保存内容の説明が変わる場合は説明バージョンを更新し、次回AI生成前に利用者へ再確認を求める。

Google OAuth設定では本番・stagingのcallback originを明示allowlistにし、ワイルドカードを使わない。リリース候補の実Google成功確認はstagingだけで行い、同じcommit SHAの証跡を残す。本番smokeはNetlify deploy metadataでSHAとoriginを再確認した後、未認証拒否と公開画面のread-only確認に限定し、実アカウントやtokenをCI artifactへ保存しない。
