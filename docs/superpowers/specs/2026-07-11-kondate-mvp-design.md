# こんだて日和 MVP 設計書

- 日付: 2026-07-11
- 状態: ユーザー承認済み
- 対象: 初回公開する実用 MVP

## 1. 目的

毎日の献立決めに悩む家庭の利用者が、ITに詳しくなくても、家族構成と冷蔵庫の食材に合った1食分の献立を短時間で決められるWebアプリを提供する。

成功条件は次のとおり。

- スマートフォンで、基本3項目を選ぶだけで献立を生成できる。
- 朝食・昼食・夕食から1食を選べる。
- 家族の年齢層、量、アレルギー、好みを毎回入力しなくてよい。
- 冷蔵庫にある消費したい食材を献立に活用できる。
- 材料、人数分の分量、手順まで確認でき、そのまま調理へ移れる。
- 献立全体または1品だけを、同じ条件で再生成できる。
- 無料AIモデルの利用を前提とし、意図しない有料モデル利用を防ぐ。
- 認証、個人上限、全体上限によってAIの濫用を抑える。

UI上の製品名には「こんだて日和」を使用する。名称変更は機能設計に影響しない独立したブランディング変更として扱う。

## 2. MVPの範囲

### 含める機能

- Google OAuthとメールのマジックリンクによるログイン
- 家族メンバーの初回設定と編集
- 朝食・昼食・夕食から1食を選んだ献立生成
- 基本条件と任意の追加条件
- 簡易冷蔵庫リストと使い切り食材の選択
- 材料、分量、調理時間、手順、家族向け取り分けメモ
- 献立全体と料理1品の再生成
- 履歴とお気に入り
- 編集可能な買い物リスト
- 1ユーザー1日5回の成功生成上限
- アプリ全体のOpenRouter呼び出し上限
- アカウントとユーザーデータの削除
- スマートフォン最適化と基本アクセシビリティ

### MVPに含めない機能

- 1日3食または7日分の一括生成
- 家族間の献立・買い物リスト共有
- 有料プラン、決済、広告
- 厳密な栄養計算、カロリー管理、治療食
- 離乳食、嚥下障害向け食、医療上の食事指導
- 在庫の自動減算、バーコード、レシート読み取り
- 期限通知のプッシュ配信
- ネイティブアプリ

## 3. 技術構成

### 3.1 採用技術

- フロントエンド: React、TypeScript、Vite、Tailwind CSS
- ルーティング: React Router
- サーバー状態: TanStack Query
- フォームと実行時検証: React Hook FormとZod
- API: TypeScriptによるNetlify Functions
- 認証・DB: Supabase Auth、Supabase PostgreSQL
- AI: OpenRouter Chat Completions API
- 単体・コンポーネントテスト: Vitest、React Testing Library
- E2E: Playwright
- DBテスト: pgTAP

SSRは使用しない。主要画面がログイン後のアプリでありSEO要件がないため、Vite SPAを採用して運用面を単純にする。

### 3.2 本番配置

- Netlify: 静的フロントエンド、Netlify Functions、環境変数
- マネージドSupabase: Auth、PostgreSQL、RLS、マイグレーション
- OpenRouter: Netlify Functionからのみ呼び出す

OpenRouter APIキーとSupabaseの秘密鍵はブラウザへ配布しない。ブラウザにはSupabaseの公開URLとpublishable keyだけを渡す。

### 3.3 ローカル環境

リポジトリ直下のDocker Composeを入口とし、次をまとめて起動する。

- Vite開発サーバーとNetlify Viteプラグインを含むアプリコンテナ
- Supabase公式Docker Compose構成のローカル用サービス群
- マジックリンク確認用のローカルメール受信サービス
- E2E用の疑似OpenRouterサービス

開発者は `docker compose up` を中心にアプリ、Auth、DB、Functionsを確認できる。ローカルSupabaseのスキーマは本番と同じSQLマイグレーションから構築する。Supabaseの自己ホスト構成はローカル開発専用であり、本番には使用しない。

## 4. システム境界とデータフロー

### 4.1 通常のデータ操作

ログイン済みブラウザはSupabaseクライアントを使い、プロフィール、冷蔵庫、履歴、買い物リストを操作する。全公開テーブルでRLSを有効にし、`auth.uid()` と `user_id` が一致する行だけを本人が操作できる。

### 4.2 AI生成

1. ブラウザが一意なidempotency keyと入力条件をNetlify Functionへ送る。
2. FunctionがSupabase JWT、入力サイズ、列挙値、同時実行状態を検証する。
3. DB関数が日本時間の日付でユーザー枠と、最初の外部呼び出しに使うアプリ全体枠を原子的に予約する。
4. Functionが生成条件をDBから読み、個人を特定しないAI入力へ変換する。
5. 外部送信の直前にアプリ全体枠を消費済みへ確定し、OpenRouterへ環境変数の優先順どおりに無料モデル一覧を渡す。
6. JSON Schemaによる構造化出力を要求する。
7. Zodと業務ルールで応答を再検証する。
8. 正常な献立と生成条件スナップショットをDBトランザクションで保存する。
9. ユーザー予約枠を成功へ確定し、結果を返す。

OpenRouterへ送信する前に処理を中止した場合は、ユーザー枠と未使用のアプリ全体予約枠を返却する。OpenRouterへ送信した外部呼び出しは、失敗してもアプリ全体上限から戻さない。

構造または業務ルールに違反した応答は、実際に応答したモデルを除外した無料モデル一覧で1回だけ自動修正する。修正呼び出しには新しいアプリ全体枠が必要だが、ユーザー枠は追加予約しない。修正枠を確保できない場合、修正後も不正な場合、タイムアウト、またはOpenRouter障害の場合は献立を保存せず、ユーザー予約枠を返却する。

AI応答は `success` または `constraint_conflict` のどちらかとする。選択した冷蔵庫食材を安全に使えない場合は `constraint_conflict` と理由、対象食材を返し、献立を保存せずユーザー枠を返却する。外部呼び出し済みのアプリ全体枠は消費済みのままとする。

### 4.3 再生成

- 献立全体: 元の生成条件スナップショットをそのまま使用する。
- 1品だけ: 元の条件、残す料理、差し替える料理の役割を渡し、献立全体との重複や不整合を避ける。
- 再生成結果は元データを上書きせず、新しい献立として保存する。
- 全体再生成と1品再生成はいずれも、成功時にユーザー上限を1回消費する。

## 5. 認証とアクセス制御

- 未ログイン利用者はログイン画面と認証コールバック以外のアプリ画面へ入れない。
- 認証方法はGoogle OAuthとメールのマジックリンクとする。
- 登録とログインを別画面にせず、「Googleで続ける」「ログイン用メールを送る」と表現する。
- 匿名ユーザーによるAI生成は許可しない。
- 1ユーザーにつき同時に1件だけAI生成を実行できる。
- すべてのユーザーデータテーブルでRLSを有効にし、別ユーザーの読み書きを拒否する。
- アカウント削除は確認操作後にサーバー側で関連データとAuthユーザーを削除する。

## 6. 家族メンバーと食事上の配慮

単純な家族人数ではなく、複数の家族メンバーを保存する。正確な生年月日は収集しない。

各メンバーは次を持つ。

- 呼び名。任意であり、AIへ送信しない。
- 年齢区分: 幼児、小学生、中高生、大人、高齢者
- 食べる量: 少なめ、普通、多め
- アレルギー
- 苦手食材
- 辛さ: なし、控えめ、普通
- 食べやすさ: 指定なし、小さめ、骨なし、やわらかめ。複数選択可

離乳食完了後の幼児からを対象とする。離乳食、嚥下障害向け食、治療食は生成しない。「やわらかめ」は一般家庭の食べやすさの希望であり、嚥下調整食ではないことを画面に明記する。

アレルギーは必須制約、苦手食材・辛さ・食べやすさは希望制約として区別する。生成結果には全体の材料総量と、必要に応じて「味付け前に子ども分を取り分ける」「高齢者分は小さく切る」などの個別メモを含める。

AI生成レシピだけでアレルギー安全を保証しない。加工品の原材料表示と家庭内の混入を利用者自身が確認する注意書きを、初回設定と生成結果に表示する。

## 7. 献立作成条件

### 7.1 常に表示する基本条件

1. 食事: 朝食、昼食、夕食
2. メイン食材: 候補チップまたは自由入力。複数指定可
3. ジャンル: 和食、洋食、中華、おまかせ

### 7.2 折りたたむ追加条件

- 調理時間: 15分以内、30分以内、45分以内、指定なし
- 予算: 節約優先、標準
- 今回だけ避ける食材
- 自由メモ。最大200文字とし、命令ではなく献立条件のデータとして扱う。

家族設定は自動反映し、この画面で一時的に人数や対象メンバーを変更できる。一時変更は保存済みプロフィールを上書きしない。

### 7.3 生成する品数

- 朝食: 基本2品
- 昼食: 基本2品
- 夕食: 主菜、副菜、汁物の3品

生成結果は料理ごとに、役割、料理名、説明、調理時間、材料、人数分の分量、番号付き手順、家族向け調整メモを含む。

## 8. 簡易冷蔵庫リスト

冷蔵庫食材は次を保存する。

- 食材名。必須
- 分量と単位。任意
- 期限日。任意
- 作成日時

期限日は利用者入力にもとづく並べ替えと注意表示にだけ使用し、安全性や可食性をアプリが判定しない。

献立作成時に、今回使いたい冷蔵庫食材を複数選択する。メイン食材は料理の中心、冷蔵庫食材は献立のいずれかで消費する必須条件として区別する。アレルギーと矛盾する食材は送信前に拒否する。

生成結果に、各冷蔵庫食材をどの料理で使うか表示する。使えない条件がある場合は無視せず、理由を返す。調理後に利用者が「使い切った」項目を選び、まとめて削除できる。AIの推測による自動減算は行わない。

## 9. 履歴・お気に入り・買い物リスト

### 9.1 履歴とお気に入り

- 正常生成された献立は自動保存する。
- 新しい順に履歴表示する。
- 献立単位でお気に入りを付け外しできる。
- 過去条件から全体または1品を再生成できる。
- 過去の履歴は利用者が削除できる。

### 9.2 買い物リスト

- 献立結果からワンタップで作成する。
- 野菜、肉・魚、乳製品・卵、乾物、調味料、その他の売り場分類を持つ。
- 同じ正規化名かつ同じ単位で安全に換算できる材料だけを合算する。
- 合算できない単位や曖昧な分量は別項目のまま残す。
- チェック、数量編集、項目追加、削除に対応する。
- 冷蔵庫にある材料を買い物対象から外す操作を用意する。

## 10. UI・UX

### 10.1 画面構成

- ログイン
- 初回家族設定
- 献立作成
- 生成中
- 献立結果
- 冷蔵庫
- 履歴・お気に入り
- 買い物リスト
- 設定・アカウント削除

下部ナビゲーションは「献立」「冷蔵庫」「履歴」「買い物」「設定」の5項目とする。

### 10.2 承認済みの視覚方針

- ホーム: 3ステップ集中型
- 献立結果: 主菜・副菜・汁物を切り替える料理タブ型
- 背景: 温かいオフホワイト
- 主操作: テラコッタ
- 冷蔵庫関連: 落ち着いた緑
- 本文: 16pxを基準とし、補足文字も12px未満にしない。
- タップ領域: 最低44px相当

料理タブは常時表示し、選択状態を色だけでなく文字、太さ、枠でも示す。アイコン単独の主要操作、スワイプ限定操作、色だけの状態表現は使用しない。

### 10.3 操作上の配慮

- ログイン画面ではGoogleを最上部、メールをその下に表示する。
- 追加条件は初期状態で折りたたむ。
- 生成ボタンの近くに「本日あとN回」を表示する。
- 再生成前に「1回使用・残りN回」を表示する。
- 生成中は段階的な日本語メッセージを表示し、二重送信を無効化する。
- 保存成功した献立は画面終了でも失われない。
- 削除は確認または取り消し機能を伴う。
- PCでは中央の最大幅を制限し、スマートフォンと同じ操作順を維持する。

## 11. AIモデル、上限、構造化出力

### 11.1 モデル設定

運営者は環境変数 `OPENROUTER_MODELS` に、優先順のモデルIDをカンマ区切りで指定する。各IDは `:free` で終わり、構造化出力をサポートする必要がある。空、重複、有料IDはローカル起動時と本番ビルド時に拒否する。本番デプロイ検証ではOpenRouter Models APIを参照し、構造化出力への対応も確認する。

OpenRouterの `models` パラメータでモデル間フォールバックを行い、provider設定で要求パラメータ対応を必須にする。無料モデルがすべて利用できない場合、有料モデルへは切り替えない。

### 11.2 上限

- `USER_DAILY_AI_LIMIT`: 初期値5。成功した通常生成、全体再生成、1品再生成を合算する。
- `GLOBAL_DAILY_AI_LIMIT`: 初期値45。NetlifyからOpenRouterへ実際に送ったHTTPリクエストを、成功・失敗を問わず合算する。
- 日付境界: `Asia/Tokyo`
- 同時実行: 1ユーザー1件

ユーザー枠は予約、成功確定、返却の状態を持つ。アプリ全体枠は予約、外部送信時の消費確定、未送信時の返却を持つ。サーバー停止で予約が残った場合に備え、期限切れ予約をDB関数が安全に解放する。外部送信後に確定したアプリ全体枠は、外部の無料枠消費を反映するため返却しない。

### 11.3 出力検証

AI応答はJSON SchemaとZodの両方で検証し、少なくとも次を確認する。

- 食事区分に合う品数と料理役割
- 正の分量または明確な家庭用単位
- 空でない調理手順
- 必須の冷蔵庫食材が料理に含まれること
- 登録アレルギーとの明白な矛盾がないこと
- 家族向け調整が元レシピと矛盾しないこと
- 医療・治療効果を断定していないこと

検証は安全性を保証するものではないため、利用者向け注意書きは維持する。

## 12. データモデル

利用者が操作する次のテーブルは `public` スキーマに置く。全行が `user_id` を持ち、RLS対象とする。

- `profiles`: ユーザー単位の表示・初回設定状態
- `household_members`: 年齢区分、量、辛さ、食べやすさ
- `member_allergies`: メンバーごとのアレルギー
- `member_dislikes`: メンバーごとの苦手食材
- `pantry_items`: 冷蔵庫食材、任意の数量・期限
- `menus`: 食事区分、ジャンル、条件スナップショット、お気に入り、元献立参照
- `dishes`: 料理の役割、名前、説明、調理時間、家族向けメモ
- `dish_ingredients`: 材料、数量、単位、売り場分類、冷蔵庫食材との対応
- `recipe_steps`: 料理ごとの順序付き手順
- `shopping_lists`: 買い物リストのメタデータ
- `shopping_items`: 分類、材料、数量、チェック、手動項目フラグ

AI制御用の次のテーブルはブラウザに公開しない `private` スキーマに置き、通常のクライアントロールにはGRANTしない。

- `ai_generation_requests`: user_id、idempotency key、種別、予約、成功・失敗、実モデル、処理時間、元献立参照
- `ai_user_daily_usage`: user_id、日本時間の日付、予約数、成功数
- `ai_global_daily_usage`: 日本時間の日付、予約数、外部送信済み数

枠の予約・確定・返却は、固定した `search_path` と最小権限を持つDB関数だけから操作する。

生成時の家族条件と入力条件は `menus` の構造化JSONスナップショットにも保存する。プロフィール変更後も履歴を正しく再現するためであり、AIの未検証な生応答や内部プロンプトは保存しない。

## 13. API境界

ブラウザから直接行う通常CRUDはSupabase RLSで保護する。秘密鍵やAIを必要とする処理だけをNetlify Functionへ集約する。

- `POST /api/generations/menu`: 新規または献立全体の再生成
- `POST /api/generations/dish`: 1品だけの再生成
- `GET /api/usage/today`: ユーザー残数と受付可否
- `DELETE /api/account`: 関連データとAuthユーザーの削除

生成APIはSupabase access token、idempotency key、Zodで検証可能なJSON本文を要求する。自由入力は長さを制限してAI入力内でデータとして明確に区切る。AIには外部ツールやデータアクセスを与えず、自由入力にかかわらず構造化出力を再検証してから利用する。

## 14. エラー処理と表示

APIは機械判定用エラーコードと利用者向け日本語メッセージを分離する。

- 個人上限: 「今日は5回利用しました。明日またお試しください」
- 全体上限: 「本日分の受付がいっぱいです」
- モデル混雑・停止: 「うまく作れませんでした。少し待ってもう一度お試しください」
- 条件矛盾: 「アレルギー食材が、使いたい食材に含まれています」
- 認証期限切れ: 再ログインを案内する。
- 通信切断: 保存済み状態を再確認したうえで再試行を案内する。
- 不正なAI応答: 破損内容を表示せず、ユーザー枠を返して再試行を案内する。

ログにはリクエストID、エラーコード、処理時間、実際のモデルIDだけを含める。氏名、メール、アレルギー、自由入力、完全なAI入力と出力はログへ残さない。

## 15. テスト方針

### 15.1 単体テスト

- 入力とAI出力のスキーマ
- 無料モデル環境変数の検証
- 家族条件とアレルギー矛盾の判定
- 買い物材料の安全な合算
- プロンプト組み立てと匿名化
- エラーコードから日本語表示への変換

### 15.2 DB・API結合テスト

- RLSが別ユーザーの全CRUDを拒否すること
- 日本時間の日次境界
- 同時実行でもユーザー上限を超えないこと
- 個人枠の予約、成功、失敗返却、期限切れ解放
- アプリ全体枠を外部送信前の中止では返し、外部送信後の失敗では返さないこと
- idempotency keyによる二重実行防止
- 全体・1品再生成の条件維持
- アカウント削除の関連データ削除

### 15.3 コンポーネント・E2Eテスト

- ログイン、初回家族設定、献立作成、冷蔵庫選択
- 生成中、結果タブ、再生成、履歴、お気に入り
- 買い物リスト、空状態、上限、通信切断
- 主要なスマートフォン幅とPC幅
- キーボード操作、フォーカス、ラベル、コントラストの自動検査

通常テストは疑似OpenRouterを利用し、外部無料枠を消費しない。実OpenRouterのスモークテストは明示的な環境変数がある場合だけ手動または限定CIで1回実行する。

### 15.4 CIゲート

静的解析、型検査、単体、DB、結合、E2E、Netlify向け本番ビルドを順に実行する。いずれかが失敗した変更はデプロイ対象にしない。本番デプロイ後は公開ログイン画面と認証必須APIの拒否応答を確認する。

## 16. セキュリティとプライバシー

- OpenRouter APIキーとSupabase秘密鍵をサーバー環境変数だけで扱う。
- RLSと最小権限のGRANTをすべての公開テーブルに適用する。
- Netlify FunctionでJWTを再検証し、クライアント送信のuser_idを信用しない。
- idempotency、同時実行制限、個人・全体上限で濫用を抑える。
- 自由入力を短く制限し、AIプロンプトでは明確にデータとして分離する。
- 家族の呼び名、メール、正確な生年月日をAIへ送らない。
- 秘密情報や食事制約をアプリログへ出さない。
- アカウント削除と履歴削除を利用者自身が実行できる。

## 17. 受け入れ条件

MVP完成は、次をすべて満たした状態とする。

1. Docker Composeからローカルアプリ、Supabase、メール確認、疑似AIを起動できる。
2. Googleまたはマジックリンクでログインし、家族メンバーを登録できる。
3. 朝昼夕、メイン食材、ジャンルから献立を生成できる。
4. 冷蔵庫食材を選び、生成結果で使用先を確認できる。
5. 材料、人数分の分量、手順、家族向け調整を料理タブで確認できる。
6. 同条件で献立全体または1品だけを再生成できる。
7. 履歴、お気に入り、買い物リストが本人にだけ保存される。
8. 成功生成が1日5回に制限され、同時実行で超過しない。
9. `:free` 以外のOpenRouterモデル設定で起動またはデプロイ検証が失敗する。
10. AI失敗時に壊れた献立を保存せず、ユーザー枠が返る。
11. 選択した冷蔵庫食材を使えない場合、理由を表示しユーザー枠が返る。
12. 主要スマートフォン幅で横スクロールせず、主要操作が44px以上の領域を持つ。
13. CIの型、単体、DB、結合、E2E、ビルドがすべて成功する。

## 18. 運用上の前提

OpenRouter無料モデルは提供状況と上限が変動する。運営者はデプロイ時に、構造化出力へ対応する現行の無料モデルIDを `OPENROUTER_MODELS` に明示する。モデル停止時は環境変数の順序を更新する。有料モデルを自動追加する処理は実装しない。

アプリ全体上限に達した場合は受付終了を表示し、翌日の日本時間に自動再開する。OpenRouter内部のモデル・プロバイダーフォールバックが複数の無料試行として扱われる場合、アプリが数える1回のHTTP呼び出しより先にOpenRouter側上限へ達する可能性がある。その場合も課金へ切り替えず、モデル混雑・停止として案内する。

公開規模が無料枠を継続的に超える場合は、有料モデルへの自動移行ではなく、運営方針を改めて決定してから別スコープで設計する。
