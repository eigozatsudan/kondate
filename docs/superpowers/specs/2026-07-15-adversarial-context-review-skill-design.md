# コンテキスト限定敵対的レビュー・修正スキル設計

## 目的

ユーザーが渡したコンテキストをレビュー境界として固定し、その範囲を敵対的にレビューする。候補指摘を独立したサブエージェントで検証し、成立した指摘だけを修正して、修正後の再レビューと検証まで完了するグローバル Codex スキルを作る。

## 配置と名前

- 配置: `${CODEX_HOME:-~/.codex}/skills/reviewing-adversarially/`
- スキル名: `reviewing-adversarially`
- 構成: `SKILL.md` と `agents/openai.yaml`
- スクリプト、参照資料、アセットは設けない。判断中心のワークフローであり、実行対象ごとの検証コマンドは対象リポジトリから決定するためである。

## 入力契約

スキルは依頼と渡されたコンテキストから、次を明示的に整理する。

- 目的と受入条件
- レビュー対象となるファイル、ディレクトリ、diff、文書、または貼り付けられた内容
- 変更を許可された範囲
- 理解のために参照のみ許可された依存範囲
- 明示された除外範囲
- 実行可能な検証コマンドと環境制約

変更可能範囲が曖昧で、安全な既定値を置けない場合だけユーザーへ確認する。依存関係の理解に必要な範囲外ファイルは読み取りできるが、変更してはならない。成立した指摘の修正に範囲外変更が不可欠なら、未解決事項として報告して停止する。

## ワークフロー

1. オーケストレーターが入力契約を「スコープ台帳」として固定する。
2. 独立 reviewer サブエージェントへ、スコープ台帳と生の対象コンテキストを渡す。reviewer は正しさ、セキュリティ、データ損失、境界条件、競合、契約違反、テスト不足を優先し、根拠箇所と再現方法を伴う候補だけを返す。
3. オーケストレーターが候補を重複排除する。スタイル上の好みや範囲外の改善提案は修正候補にしない。
4. verifier サブエージェントへ、対象コンテキストと正規化した候補指摘を渡す。先行 reviewer の推論過程や期待する結論は渡さない。verifier は各候補を成立、棄却、未確定に分類し、静的トレース、最小再現、既存テスト、または対象に適した検証で根拠を示す。
5. fixer サブエージェントは成立した指摘だけを、変更可能範囲内で最小修正する。可能な場合は先に失敗を再現するテストを追加し、対象リポジトリの指示に従う。
6. fresh re-reviewer サブエージェントが修正後の対象と受入条件を独立に再レビューする。修正内容の正当化や「直ったはず」という結論は渡さない。
7. オーケストレーターが関連テストと最終検証を実行する。新しい成立指摘があれば verifier と fixer のループへ戻る。
8. すべての成立指摘が解消され、検証が通った場合だけ完了とする。

サブエージェントを利用できない環境では、独立検証を偽装せず、その制約を明示してユーザーへ判断を求める。

## サブエージェント分離

- reviewer は候補発見に専念し、コードを変更しない。
- verifier は候補の真偽判定に専念し、コードを変更しない。
- fixer は検証済み指摘だけを変更する。
- re-reviewer は修正後成果物を新規の対象として扱う。
- 同じサブエージェントへ発見、検証、修正、合格判定を兼任させない。
- 並列 reviewer を使う場合は、対象を重複させるのではなく、独立した脅威モデルまたは明確に分離できる対象へ割り当てる。

## 指摘の品質契約

各指摘は次を満たす。

- 重大度と影響
- 根拠となる対象箇所
- 具体的な失敗経路または反例
- 再現・確認方法
- スコープ内で修正可能かどうか

証拠が不足する候補は推測で修正しない。未確定として、追加で必要な証拠を明示する。誤検知と判断した候補も、棄却理由を最終報告に残す。

## エラー処理と停止条件

- 対象が読めない、必要な依存が欠ける、検証環境が利用できない場合は、可能な静的確認を完了して制約を報告する。
- 破壊的操作、外部システムへの書き込み、新しい権限、範囲外変更が必要なら、実行せずユーザーの承認を求める。
- テスト失敗が修正と無関係に見えても無視せず、既存失敗か回帰かを切り分ける。
- fixer と re-reviewer の往復には上限を設けず、新しい根拠がある間は継続する。同じ阻害条件が続く場合は、実行環境のブロック規約に従う。

## 最終出力

最終報告は次の順で簡潔に示す。

1. 完了判定とレビュー範囲
2. 検証済み指摘と実施した修正
3. 棄却した候補と理由
4. 実行した検証と結果
5. 未解決事項、範囲外事項、残余リスク

変更が不要だった場合も、レビュー対象、検証方法、指摘なしと判断した根拠を示す。

## 評価計画

実装前のベースラインと実装後の評価には、最低3つのシナリオを使う。

1. 対象範囲内に実在する不具合と、もっともらしい誤検知を混在させる。
2. 修正に範囲外ファイルの変更が必要に見える状況を与え、境界を守れるか確認する。
3. reviewer の候補に権威性と時間圧を加え、verifier が未検証の指摘をそのまま修正しないか確認する。

ベースラインではスキルを渡さず、候補の鵜呑み、スコープ逸脱、自己検証による早期完了などの実際の失敗を記録する。実装後は同じシナリオを新しいサブエージェントへ渡し、成立指摘のみの修正、境界維持、独立再レビュー、検証結果の明示を確認する。

## 完了条件

- スキルがグローバル配置され、構造バリデーションを通る。
- `agents/openai.yaml` が SKILL.md と整合する。
- ベースライン評価の失敗を SKILL.md が具体的に防ぐ。
- 実装後の3シナリオで、検証前修正、スコープ逸脱、独立再レビュー省略が発生しない。
- fresh re-reviewer が見つけた新しい成立指摘を反映し、再評価が通る。
