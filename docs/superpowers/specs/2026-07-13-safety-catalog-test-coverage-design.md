# 安全カタログテスト強化設計

## 目的

`supabase/tests/database/02_safety_catalogs.test.sql` が総件数だけでなく、規制区分、食品安全ルール、重要な別名の破損を検出できるようにする。

## 検証範囲

- `jp-caa-2026-04.v1` のアレルゲンを必須9件、推奨20件、合計29件として厳密に検証する。
- `jp-caa-child-shape-2026-07.v1` の7つのルールIDを完全集合として検証し、不足と余分な行の両方を検出する。
- 直接・派生・加工別名のうち、安全判定を代表する組を `VALUES` 表で定義し、アレルゲンID、正規化別名、種別、表示確認要否の完全一致を検証する。
- シード全体をテストへ複製せず、総件数と代表ケースの組み合わせで分類ミスと主要な意味変更を検出する。

## 実装方法

既存の権限、主キー、RLS、年齢・形状ルールのアサーションは維持する。件数中心のアサーションを次の検証へ置き換える。

1. 規制区分別の正確な件数を `is(...)` で検証する。
2. 期待する7ルールを `VALUES` で宣言し、期待集合と実データの双方向 `except` が空であることを検証する。
3. 重要別名を `VALUES` で宣言し、対応行が完全一致することを検証する。

重要別名には、少なくとも直接別名、ラベル確認不要の派生別名、複数アレルゲンへ紐づく加工品を含める。

## 検証方法

強化したアサーションが実際に破損を検出することを確認するため、一時的にローカルDBの分類、ルールID、重要別名をそれぞれ変更し、focused pgTAP が期待した理由で失敗することを確認する。一時変更はトランザクションでロールバックする。その後、実データに対して focused pgTAP と `npm run db:test` 全体を実行し、すべて成功することを確認する。

## 対象外

- 安全カタログの本番データ変更
- マイグレーションやテーブル定義の変更
- 全別名のテストコードへの完全複製
