# 提出前E2E既存不具合修復 設計

## 背景

Playwright MCPセットアップの提出前必須検証で、`./scripts/run-e2e.sh` が18件成功、4件失敗となった。4件は同じ2テストのdesktop/mobile実行であり、merge base `203df0c` から現在のブランチまでの変更対象には、失敗した家族設定、期限切れ食材選択、対象E2Eの該当行、DB制約、CSSは含まれていない。

失敗は今回のPlaywright MCP変更による回帰ではないが、リポジトリの必須検証を完了するには、既存の段階的な確認フローとE2E・保存処理の契約不整合を修復する必要がある。

## 根本原因

### 期限切れ食材の確認

期限切れ食材を選ぶと、アプリは直ちに選択を確定せず、確認dialogを開いてcheckboxを未選択かつ一時的に無効な状態へ戻す。E2Eはこの操作にPlaywrightの`check()`を使用しているため、click自体が成功してdialogが開いても、最終状態がcheckedではないとして失敗する。

下部ナビゲーションによるpointer eventの遮断は最初のretryだけであり、その後のclickとdialog表示は成功しているため根本原因ではない。

### 完了済み家族のアレルギー登録

完了済み家族でアレルギー状態を`registered`へ変更すると、現在の画面はアレルギーを追加する前に家族行だけを保存する。DBは`complete`かつ`allergy_status = 'registered'`の家族に1件以上のアレルギーを要求するため、HTTP 400、SQLSTATE `23514`、`member_registered_allergy_required`で保存を拒否する。

初回設定中のdraft家族では中間状態が許可されるが、complete家族には同じ保存順序を使用できない。

## 採用方針

確認やDB整合性を伴う操作を、単一入力の即時確定ではなく二段階の遷移として扱う。

1. 期限切れ食材のE2Eは`click()`で確認dialogを開き、dialog表示中は未選択、利用者の確認後は選択済みになることを既存の検証で確認する。
2. complete家族をアレルギー0件の状態で`registered`へ切り替えた場合、画面の入力値だけを更新し、家族行の保存は保留する。
3. 保留中は「登録ありの場合は1つ以上選んでください」と表示し、DBへ失敗することが分かっているPATCHを送らない。
4. 最初の標準または自由登録アレルギーの追加に成功した後、同じ家族の`allergy_status = 'registered'`を保存する。
5. 家族行の保存成功後に既存の安全条件無効化を実行し、「最新条件で再確認します」の成功表示へ遷移する。

アレルギー行の追加後に家族行を保存する順序でDB制約を満たせるため、新しいRPCやマイグレーションは追加しない。

## データフロー

complete家族で`none`から`registered`へ変更した場合の流れは次のとおり。

```text
select registered
  -> ローカルフォーム値を更新
  -> アレルギー0件なら家族行PATCHを保留
  -> 利用者へ1件以上の登録が必要と表示
  -> 標準または自由登録アレルギーをINSERT
  -> アレルギーqueryを更新
  -> 保留していた家族行PATCHを実行
  -> 安全条件に依存するqueryを無効化
  -> 最新条件での再確認メッセージを表示
```

既にアレルギーが1件以上ある場合や、`none`・`unconfirmed`へ変更する場合は、既存どおり入力変更時に保存する。draft家族の初回設定フローも変更しない。

## エラー処理

- アレルギー追加に失敗した場合は家族行を`registered`へ保存しない。
- アレルギー追加後の家族行保存に失敗した場合は、既存の保存失敗表示を維持し、成功メッセージを表示しない。
- 既存の保存queueを利用し、別の家族設定変更との順序を崩さない。
- 最後のアレルギー削除を禁止する既存動作とDB制約は変更しない。
- DBスキーマ、RLS、マイグレーション、アレルギー安全判定は変更しない。

## テスト

### Unit

`HouseholdSettingsForm`について次をREDから追加する。

1. complete家族でアレルギー0件のまま`registered`を選択しても`updateMember`を呼ばず、1件以上必要という表示を行う。
2. その後に標準アレルギーを追加すると、INSERT成功後に`allergy_status = 'registered'`を含む`updateMember`を呼び、安全条件を無効化する。
3. アレルギー追加が失敗した場合は`updateMember`を呼ばない。

### E2E

- 期限切れ食材の初回操作を`click()`へ変更し、既存のdialog、未選択、確認後選択の検証を維持する。
- complete家族では、`registered`選択直後に1件以上必要という案内を確認し、アレルギー追加後に最新条件での再確認メッセージを確認する。
- 修正後は失敗した2ケースをdesktop Chromiumでfocused実行し、その後、提出前検証の失敗ステップである全E2Eから順番に再実行する。

## 不採用案

### E2Eだけを成功させる

期待順序だけを変えると、画面上は`registered`でもDB上は以前の状態のままになり得る。利用者の設定が再読み込み後に失われるため採用しない。

### DB RPCでアレルギー追加と状態変更を原子的に行う

強い原子性は得られるが、RPC、権限、型生成、pgTAPを追加する必要があり、今回判明した2段階UIの修復として変更範囲が大きい。既存制約を満たす操作順で解決できるため採用しない。
