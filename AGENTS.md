# AGENT.md

このファイルは、AIエージェントがこのリポジトリで作業する際の指示書です。以下の指示に厳密に従ってください。

---

## 1. プロジェクト概要

- **目的**: 家庭料理を作る一般ユーザー（主婦など）向けの、メニュー（献立）作成 WebApp。
- **本番デプロイ先**: フロントエンドは Netlify、バックエンド/DB は Supabase。
- **想定利用者の技術レベル**: 非エンジニア。UI・文言は平易でわかりやすくすること。

---

## 2. 実行環境の前提

- **Node バージョン**: 24（`package.json` の `engines` を参照。異なる場合は `engines` を正とする）。
- **コマンド実行方式**: Node/npmコマンドだけを、原則 **Docker 経由** で実行する。ホスト上に直接 Node をインストールして実行しないこと。Codex CLI、`rg`、Git、その他のホストコマンドは、指定されたコマンドをそのままホストで実行する。
- **依存関係のインストール**: `docker compose run --rm --no-deps app npm ci`
- **サブエージェントの活用**: コマンド実行やレビューなど、独立して進められる作業は積極的にサブエージェントへ委譲する。

### コマンド実行の重要ルール

1. **コマンドを結合しない**: 権限確認のプロンプトを避けるため、Docker コマンドやホスト側コマンド（`git diff --check` など）を `&&` や `;` で連結せず、**1コマンド＝1ツール呼び出し**で独立して実行する。
2. **モデルの切り替え**: テスト実行など推論をほとんど必要としない定型処理では、**より軽量なモデル**に切り替える。レビューや設計判断など高度な推論が必要な処理では、**より高性能なモデル**への切り替えを検討する。

## 3. よく使うコマンド

すべて `docker compose run --rm --no-deps app <command>` の形式で実行する（明記あるものを除く）。

| 目的 | コマンド |
| --- | --- |

| 本番ビルド（TypeScript チェック込み） | `npm run build` |

### スタック操作（Docker 直接実行）

| 目的 | コマンド |
| --- | --- |

## 4. 実装の進め方

1. 実装は **Plan 単位**で管理し、各 Plan 内の **Task を1つずつ順番に**進める。
2. **各 Task 完了ごとに**、以下を必ず実施する:
   - `/compact` でコンテキストを圧縮する。
   - 直前の Task の変更が次の Task に悪影響を与えていないか、**コードベースと設計書を照合してレビュー**し、必要なら修正する。
3. 設計書に記載のない仕様変更を勝手に行わない。判断に迷う場合は設計書を正とし、設計書自体の不備が疑われる場合は明示的に指摘する。

---

## 5. サブエージェント運用

- 詳細なTask実行順序、引き継ぎ、レビュー判定は `SubAgents.md` を正とする。
- 親エージェントは設計、仕様判断、委譲範囲の決定、結果の統合、最終判断を担当する。
- 親のlive runtime permission overrideはカスタムエージェントの既定権限を上書きし得る。`explorer`、`reviewer`、`fast-worker` を起動する前に、親はlive runtime permissionとしてread-onlyを選択する。利用中のsurfaceで選択できない場合、その役割の編集禁止は技術的な権限境界ではなく指示上の制約に留まることを報告する。
- コードベースや設計書の読み取り調査には、読み取り専用の `explorer` を使用する。live overrideで書き込み可能になっていても、リポジトリの編集を拒否させる。
- Node/npmによる定型テスト、型チェック、Lint、フォーマット検証と、指定されたCodex CLI、`rg`、Git等のホストコマンドの実行・ログ要約には、読み取り専用の `fast-worker` を使用する。これは `SubAgents.md` の Verifier 役であり、live overrideで書き込み可能になっていてもリポジトリファイルの編集を拒否させる。
- Taskのコード変更には `implementer` を使用する。single-writerはCodex設定による役別同時数の技術的制限ではなく、親が維持するオーケストレーション不変条件である。親はImplementerを起動する前にactive agent threadを確認し、既存Implementerが完了またはcloseされるまで2体目を起動してはいけない。active状態を確認できない場合は起動を停止し、その制約を報告する。
- 設計適合性、セキュリティ、敵対的入力、境界条件、回帰、テスト不足の確認には、読み取り専用の `reviewer` を使用する。live overrideで書き込み可能になっていても、リポジトリの編集を拒否させる。
- 独立した読み取り作業だけを並列化し、サブエージェントの報告は親エージェントが根拠を確認してから採用する。
- 一次レビューと、その指摘を深掘りする二次検証には、コンテキストを共有しない別々の Reviewer エージェントを使用する。この二次検証は、Dockerコマンドを再実行する Verifier 役の検証とは別である。
- モデルまたはカスタムエージェント種別を実行環境で指定できない場合は、暗黙に指定できたと仮定せず、その制約と実際に使用した代替手段を報告する。

## 6. コーディングスタイルと命名規則

### フォーマット
- インデントは **2スペース**。
- **二重引用符**、**セミコロンあり**、**Prettier** のフォーマットに準拠。

### TypeScript
- 厳密かつ明示的な型付けを推奨する。
- `any` および安全でないキャストは避ける。

### 命名規則
- React コンポーネント / 型: `PascalCase`
- 変数 / 関数: `camelCase`
- テストファイル: `*.test.ts` / `*.test.tsx`

### インポート
- 設定済みのパスエイリアス `@/` および `@shared/` を適切な箇所で使用する。

### コメント（重要）
- コード内のコメントは**必ず日本語**で書く。
- 「何をしているか」だけでなく、**背景・意図・設計上の判断理由**を具体的に記述し、後から人間がレビューしやすいようにする。

## 7. テストガイドライン

- テストは**対象コードの近くに配置**し、焦点を絞る。
- テストの種類と用途:
  - **Vitest / React Testing Library**: ユニット・コンポーネントテスト。
  - **Playwright**: ユーザーフロー（E2E）テスト。
  - **pgTAP**: スキーマ / RLS 動作のテスト。
- テストは**決定論的**に保つ。ランダム性や時刻依存で結果が変わらないようにする。
- 対象の挙動を変更した場合は、関連テストも**必ず更新**する。

---

## 8. 変更提出前の検証フロー（必須）

変更を提出する前に、以下を**それぞれ独立したコマンド**として、この順番で実行する（`&&` 等で連結しない）。すべてがパスすることを確認する。

【注意】承認不要として許可されたリポジトリ内スクリプトを実行する前に、そのスクリプトまたは呼び出し先に未確認の変更がある場合は、差分を確認し、破壊的操作・外部送信・シークレット参照などがないことを確認する。

1. `docker compose run --rm --no-deps app npm run format:check`
2. `docker compose run --rm --no-deps app npm run lint`
3. `docker compose run --rm --no-deps app npm run typecheck`
4. `docker compose run --rm --no-deps app npx vitest run`
5. `./scripts/reset-local-db.sh`
6. `docker compose --profile test run --rm db-test`
7. `./scripts/run-e2e.sh`
8. `docker compose run --rm --no-deps app npm run build`
9. `git diff --check`

いずれかが失敗した場合は、原因を特定・修正してから、失敗したステップ以降を再実行する。

---

## 9. レビュー

レビューは**サブエージェントを使って**、以下の観点で実施する。

1. **一次レビュー**: 設計書との整合性、セキュリティ上の問題、**敵対的レビュー**（悪意ある入力・境界条件・想定外の使われ方）の観点で検証する。
2. **二次検証**: 一次レビューとは**別のサブエージェント**を使い、指摘内容をより深く検証する。指摘が妥当であれば修正する。
3. この「レビュー → 別エージェントによる深掘り検証 → 修正」のフローを連続して行う。
4. レビュー時は、より適切な（高性能な）モデルへの切り替えを検討する。

## 10. コミット / プルリクエストのガイドライン

### コミットメッセージ
- **必ず日本語**で書く。
- **Conventional Commits 形式**に従う。例:
  - `feat: 献立生成を追加`
  - `fix: 認証状態の復元を修正`
  - `chore: 開発設定を整理`

### ファイル名
- 短く、変更内容がわかる表現にする。

### プルリクエスト
- 以下を記載する:
  - 変更内容の説明。
  - 実行した検証コマンド（第7章のどれを実行したか）。
  - 関連する Issue や設計資料へのリンク。
- **UI 変更にはスクリーンショットを添付**する。
- マイグレーション、環境変数、セキュリティへの影響、ローカルスタックの変更を伴う場合は、その内容と影響範囲を明記する。

## 11. セキュリティと構成

- **シークレットや生成されたローカル認証情報を絶対にコミットしない。**
- OpenRouter などのサービスのシークレットは、**サーバー側の環境変数**に保持する。クライアントには公開しない。
- Supabase クライアントには、**目的に必要なクライアント構成のみ**を公開する。
- マイグレーションと RLS ポリシーは**本番環境に影響するもの**として扱い、スキーマ変更は必ず**ローカルの Docker スタックで検証**してから提出する。
